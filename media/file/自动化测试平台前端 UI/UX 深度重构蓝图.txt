自动化测试平台前端 UI/UX 深度重构蓝图 (v3.0 全模块实战版)

0. 架构师导言：从“大杂烩”到“极客工作台”

当前系统的前端（基于 l-vue-ui-master）虽然功能齐全，但带有极强的“通用后台管理系统（Admin Template）”的臃肿感。专业测试工具（如 Postman, Apifox, VS Code）的 UI 灵魂在于：信息分层、核心工作区最大化、无缝的键盘交互、红绿状态强感知。

以下是针对全平台 6 大核心模块的详细 UI 优化拆解指南。

1. 主控台模块 (Dashboard)

定位：测试指挥中心，一眼看透当前系统质量与资源状态。
目标文件：src/views/home/index.vue 及相关组件

现状痛点：充斥着 KoiTradeChart, KoiTimeline 等电商/通用模板的无用组件，信息噪音极大。

重构方案：

顶部 (全局概览区)：移除现有的花哨图表。改为 4 个纯净的数据大卡片（Data Cards）：

今日运行任务数 (及通过率环形图)

当前正在占用的云真机数

累计沉淀测试用例数

待处理的告警/失败数

左侧核心区 (70% 宽度)：“我的排障台”。直接放一个精简版的列表，展示最近 24 小时内运行失败的任务，附带醒目的红色 [Fail] 标签，点击直接跳转到对应的 report 页面。

右侧辅助区 (30% 宽度)：“快捷发射台”。以图标+文字的网格形式，放置最高频的动作：[+ 新建需求], [+ 触发API巡检], [+ 借用设备]。

2. 实用工具模块 (Utility Tools)

定位：开发与测试过程中的顺手小工具，要求“即用即走，零延迟”。
目标文件：src/views/utility/encryption.vue, json_parse.vue, time_change.vue

现状痛点：传统的表单模式，输入后还要找“转换/提交”按钮，效率低下。

重构方案：

交互革命：响应式实时预览。去掉所有的【转换/解析】按钮。利用 Vue3 的 watch 或 computed，左边输入/粘贴框内容一变，右边立刻实时展示解析/加解密结果。

双栏等分布局 (Splitpane)：左侧为 Input，右侧为 Output。

极客操作：右侧结果面板的右上角，必须提供一个极其醒目的 【一键复制】 图标。如果解析 JSON 失败，右侧边框应泛红并提示具体的 Syntax Error 位置。

远期进阶：考虑将这些工具做成一个全局快捷键唤醒的右侧全局抽屉 (Global Drawer)，这样用户在写用例时不需要跳出当前页面就能完成时间戳转换。

3. 接口自动化模块及子模块 (API Automation)

定位：高度结构化的数据调试与编排中心。
目标文件：src/views/api_view/api_project.vue, api_code.vue, api_params.vue 等

现状痛点：api_project.vue 是一个包含近 2000 行代码的巨石页面，所有配置都在一页上下滚动，一旦参数多，用户极易迷失。

重构方案 (Postman 化)：

全屏三段式布局 (Three-Pane Layout)：

左侧 (20%)：接口目录树 (Tree) 和搜索框。

中上 (编辑区)：剥离长表单，强制使用 Tabs 切换：[Headers] | [Query] | [Body] | [Assert 断言] | [Extract 提取]。

中下 (响应区)：默认收起，点击右上角的【Send/发送】按钮后，该区域向上弹起，展示 HTTP 状态码（200 绿色，4xx/5xx 红色）和 JSON 响应体。

动作固定：页面右上角绝对固定两个按钮：【保存 Ctrl+S】和【运行 Ctrl+Enter】。

4. Web管理模块及子模块 (Web Automation)

定位：UI 自动化的编排与元素管理，要求强关联、易拖拽。
目标文件：src/views/web_view/web.vue, element_manager.vue, web_report.vue

重构方案：

web.vue (用例编排) 的解耦手术：

严禁在中间列表直接编辑步骤！中间只展示步骤列表 (List)（支持拖拽排序）。

点击某一步骤时，从右侧滑出 Drawer (步骤配置抽屉) 进行详细参数编辑（如选择定位器、输入文本）。这样用户能始终保持对整个用例上下文的感知。

element_manager.vue (元素库)：

将单调的表格优化为“卡片+列表”双视角。对每个元素定位器，增加一个 Type 标签的高亮（如深蓝色的 XPATH，紫色的 CSS）。

web_report.vue (排障报告)：

绝对刚需：报告顶部加一个 【仅显示失败步骤】的 Switch 开关。

失败的步骤展开后，必须左右并排展示两张图：[期望状态截图 (如有)] vs [实际崩溃截图]。

5. App管理模块及子模块 (App Automation)

定位：设备与移动端任务的调度中心。
目标文件：src/views/app_view/app.vue, device_install.vue, src/views/device_view/device.vue

重构方案：

device.vue (设备大厅)：

放弃顶部的下拉框筛选，改用极其直观的状态指示 Tabs：[全部] | [🟢 空闲可用] | [🟡 占用中] | [🔴 离线故障]。

卡片化展示设备，电量低或离线状态要有呼吸灯或标红警示。

交互优化：点击设备的弹窗底部必须有 Fixed Footer (固定操作栏)，保证无论列表多长，【连接设备】、【重启】按钮永远在最下方触手可及。

app_run.vue (App 执行控制台)：

左侧流式步骤，右侧实时展示 Appium/Poco 的 Console Log。在 Log 面板增加局部搜索框和一键清屏功能。

6. 测试设计模块及子模块 (Test Design)

定位：基于 AI 的大批量数据吞吐与转化引擎（你的核心亮点）。
目标文件：src/views/requirement_view/, testcase_view/generate.vue, automation_draft_view/

现状分析：这是你 003 到 008 规范中的核心链路（PRD -> 用例 -> 自动化草稿）。

重构方案 (硬核中控室概念落地)：

业务流水的视觉连贯性：采用向右推进的布局（Pipeline UI）。

generate.vue (AI 生成区) 强制应用暗黑模式 (Dark Mode) 与全键盘操作。

左侧控制台：PRD 输入区与参考 Excel 上传。

右侧微调区：引入 vxe-table 或高配表格。禁用分页，开启虚拟滚动 (Virtual Scroller)。

极客交互：支持上下左右方向键移动焦点，敲击 Enter 直接编辑当前单元格内容（尤其是修改 AI 生成的错漏变量），支持类似 Excel 的框选和批量复制粘贴。

自动化草稿 (automation_draft_view)：

提供一个【一键转化校验】视图。左边是用例文本，右边是系统尝试生成的底层 Python/YAML 脚本草稿。对未能成功转化为脚本的“自然语言步骤”使用黄色高亮，提示用户需手动介入补齐。

7. 架构师的行动指令 (给 AI 辅助编程工具)

如果你准备开始动手，请严格按照以下顺序命令你的 AI 编码助手，防止系统崩溃：

先做减法：把这段 PRD 的第一节发给它，命令它重写 src/views/home/index.vue，删除所有非测试相关的 E-charts 组件，换成纯净的数据卡片。

再搞隔离：挑最痛的 src/views/web_view/web.vue。命令它：“请将这个庞大的 Vue 文件拆分，把编辑步骤的长表单提取出来，做成一个独立的 RightDrawer 组件，用 v-model:visible 控制它的划出”。

后攻高精尖：最后处理你的 AI 测试生成页 (generate.vue)，命令它引入 vxe-table，并配置 keyboard-config 开启全键盘编辑模式和暗黑主题。
