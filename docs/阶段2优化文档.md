《修正版阶段 2 总路线图（纳入 AI 配置 / Prompt / 需求分析 / 用例生成 / AI 助手）》

这版会把你刚才提醒的几条都正式纳入主线：

大模型 API 配置

Prompt 配置

上传需求分析内容

AI 自动生成测试用例

AI 助手

后续自动化草稿与资产落地

也就是说，阶段 2 不再只是“测试设计对象收口”，而是：

阶段 2 的新定义

把测试设计对象层、AI 生成链路、自动化桥接层，收成一条完整的业务闭环。

一、修正版阶段 2 的总结构

我建议你把阶段 2 改成这 5 段：

2.1 测试设计资产收口

先把项目 / 需求分析对象 / 测试用例这些设计资产层收稳。

2.2 AI 生成链路与 AI 助手收口

把模型配置、Prompt 模板、需求输入、AI 分析、AI 用例生成、人工确认、AI 助手辅助、调用留痕，收成一条清晰链路。

2.3 自动化草稿中心

把测试设计结果进一步转成自动化草稿，形成桥接层。

2.4 草稿到目标资产落地

让自动化草稿真正落到 API / Web / App 资产。

2.5 设计资产到执行资产追踪闭环

把项目 / 用例 / 草稿 / 自动化资产 / 执行结果串起来。

二、为什么现在必须把 AI 线纳入主线

因为你现在这个系统已经不是一个“纯手工测试平台”，而是明显具备了：

AI 模型配置

Prompt 配置

AI 需求分析

AI 自动生成测试用例

AI 助手入口

如果阶段 2 不把这条线放进去，后面会出现两个问题：

问题 1

业务闭环只写成：

项目 → 用例 → 草稿 → 自动化

但实际上你的平台最值钱的起点之一已经变成：

需求输入 → AI 生成 → 用例资产

问题 2

AI 配置、生成页、助手页、用例资产页会各自为政。
用户会看不出：

这些是不是同一套链路

当前用的哪个模型

用的是哪个 Prompt

生成结果是否可信

失败后怎么处理

所以你刚才那个提醒非常关键：
阶段 2 必须把 AI 生成链收进来。

三、修正版阶段 2 的主链路

我建议你把整个平台后续的“主故事线”明确成这个：

需求输入 / 项目建立

→ AI 分析
→ AI 生成测试用例
→ 人工确认 / 编辑
→ 保存为测试设计资产
→ 生成自动化草稿
→ 落到 API / Web / App 资产
→ 执行
→ 结果
→ 回链项目 / 用例 / 草稿

这条线才是你平台真正的差异化价值。

四、2.1 和 2.2 的关系要重新理解

这两个不是互相替代，而是：

2.1 是资产层

把“项目 / 需求分析对象 / 测试用例对象”收稳。

2.2 是生成层

基于这些对象，建立：

模型配置

Prompt 模板

AI 生成

AI 助手

人工确认

留痕

所以可以这样记：

2.1 解决：对象是什么

2.2 解决：对象怎么被 AI 生成出来

五、2.1 测试设计资产收口（修正版理解）

这一段我还是建议先做，但要明确把“需求分析对象”也纳入。

2.1 的对象层建议
第一层：测试设计项目 / 需求分析项目

这是源头对象。

它应该承接：

项目名

描述

需求来源

上传的需求内容/摘要

状态

最近更新时间

用例数

AI 生成摘要位

自动化摘要位（后续）

第二层：测试用例

这是设计产物对象。

它应该承接：

所属项目

标题

优先级

类型

步骤 / 预期

来源方式（人工 / AI）

评审状态

自动化状态位

最近更新时间

第三层：执行对象

先作为下游对象预留，不在 2.1 重做。

2.1 的重点

项目详情和用例关系明确

用例详情和项目关系明确

AI 来源位预留

自动化状态位预留

详情页开始具备“下一步入口位”

也就是：
2.1 先把来源对象收干净。

六、2.2 AI 生成链路与 AI 助手收口

这是修正版阶段 2 里新增的主段，而且非常关键。

2.2 的目标

把当前系统里分散存在的 AI 能力，统一成一条可理解、可配置、可追踪、可确认的业务链。

这条链应该至少包含：

模型 API 配置

Prompt 模板配置

需求输入 / 上传

AI 分析

AI 测试用例生成

人工确认 / 编辑

保存为测试用例资产

AI 助手嵌入式辅助

调用记录 / 失败反馈

2.2 要解决的核心问题
1）AI 配置不再只是“设置页”

现在如果 AI 配置只是放在配置中心，它只是一个后台设置。
但你真正需要的是让用户知道：

当前生成用的是哪个模型

当前场景用的是哪个 Prompt

哪些参数是这个场景可调的

失败后怎么处理

所以 2.2 要把 AI 配置从“静态设置”提升成：
业务生成链路的底座。

2）需求输入 → AI 生成 → 确认保存 这条链产品化

这一段建议明确变成一个流程，而不是“页面里放个生成按钮”。

建议流程页
第一步：选择或新建测试设计项目
第二步：输入或上传需求内容
第三步：选择模型 / Prompt 模板
第四步：运行 AI 分析
第五步：预览生成测试用例
第六步：人工确认 / 编辑
第七步：保存为测试用例资产

这才是你平台现在最该产品化的 AI 生成链。

3）AI 助手不要只做独立聊天页

AI 助手应该分两类理解：

平台级 AI 助手

比如：

解释平台功能

帮用户找页面

提供引导

这个可以继续保留，但不是阶段 2 主线。

业务域 AI 助手

这个才是阶段 2 主线，要嵌入到：

需求分析页

用例生成页

用例详情 / 编辑页

后续自动化草稿页

也就是说，AI 助手应该嵌在流程里，不只作为独立入口。

4）AI 调用要有留痕

这一点非常重要。
平台后面会越来越需要知道：

这个用例是不是 AI 生成的

生成时用的是哪个模型

生成时用的是哪个 Prompt

生成失败原因是什么

最后是谁确认保存的

这就是 AI 治理的第一层。

2.2 页面建议
A. AI 配置页（收口）

要明确场景：

需求分析模型

用例生成模型

助手模型（如分开）

B. Prompt 模板页

要明确模板和场景关系：

需求分析模板

用例生成模板

助手模板

C. 需求输入 / 分析页

这是 AI 生成链的起点。

D. 用例生成确认页

这是 AI 生成链里最关键的“人工确认层”。

E. 用例详情 / 编辑页中的 AI 来源摘要

让用户知道这条用例的来源。

2.2 最终结果

做完以后，你应该能清楚讲出：

模型怎么配

Prompt 怎么配

需求怎么输

AI 怎么生成

结果怎么确认

用例怎么入库

哪些地方 AI 助手能帮忙

调用是否可追踪

这就是 2.2 的价值。

七、2.3 自动化草稿中心

这一段和我之前讲的一样，但现在位置更清晰了：

它不是紧跟“人工用例”，而是跟在：

人工 / AI 生成的测试用例资产之后。

也就是：

项目/需求输入
→ AI 生成或人工整理
→ 用例资产
→ 自动化草稿中心

所以自动化草稿中心的来源应该既支持：

用例

也可能支持项目级批量草稿生成（后续）

八、2.4 草稿到目标资产落地

这一段不变，还是：

草稿到 API

草稿到 Web

草稿到 App

但现在它有更清楚的来源链：

需求 / 项目 → AI 生成 → 用例 → 草稿 → 目标资产

这会让后面的来源追踪更自然。

九、2.5 设计资产到执行资产追踪闭环

这一段也不变，但现在要明确再加一层来源信息：

执行结果页以后最好能看到：

来源项目

来源用例

用的草稿

生成来源是否来自 AI

也就是说，阶段 2 收口后，平台就能逐渐回答：

这次执行失败的对象，最初来自哪个需求/项目，是否由 AI 生成，用的是哪份草稿。

这会很有价值。

十、修正版阶段 2 的推荐执行顺序

我建议顺序改成这样：

第 1 步：2.1 测试设计资产收口

先把项目 / 需求分析对象 / 用例对象层收稳。

第 2 步：2.2 AI 生成链路与 AI 助手收口

把模型、Prompt、需求输入、AI 生成、确认保存、助手嵌入收成一条链。

第 3 步：2.3 自动化草稿中心

开始桥接到自动化层。

第 4 步：2.4 草稿到目标资产落地

真正进 API / Web / App。

第 5 步：2.5 设计资产到执行资产追踪闭环

最后把来源链补齐。

十一、修正版阶段 2 的风险控制
2.1 风险

容易只是在修页面，不是在收对象关系。
所以一定要盯住：对象层级、来源关系、摘要字段。

2.2 风险

最容易把 AI 能力继续分散在：

配置中心

生成页

助手页
各自为政。
必须强制收成一条链。

2.3 风险

最容易把草稿系统散塞进各自动化模块。
一定要单独建桥接域。

2.4 风险

最容易一上来追求“智能全自动”。
第一版先把保存和跳转做稳。

2.5 风险

最容易加一堆追踪字段，结果没有最小闭环。
先做最少可用追踪。

十二、修正版阶段 2 完成后，你的平台会变成什么

如果修正版阶段 2 做好了，你的平台就不只是：

有测试设计

有 AI 生成

有自动化

有执行

这四块并存，

而会变成真正一条可讲清楚的产品链：

需求输入 / 项目

→ AI 分析
→ AI 测试用例生成
→ 人工确认
→ 用例资产
→ 自动化草稿
→ 自动化资产
→ 执行
→ 结果
→ 回链来源

这就是非常完整、也非常有差异化的一条测试平台价值链。

十三、我建议你下一步怎么做

现在最合理的下一步不是先写 2.2 提示词，
而是先把：

2.1 测试设计资产收口

单独做成可执行任务。

因为只有来源对象收稳了，2.2 的 AI 生成链路才不会悬空。
