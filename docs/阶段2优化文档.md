阶段 2 的新定义

阶段 2 的目标不再只是“测试设计资产化”，而是：

把测试设计对象层、AI 生成链路、自动化桥接层、执行追踪层收成一条完整业务闭环。

也就是让平台从：

有测试设计

有 AI 能力

有自动化模块

有执行模块

这些并列能力，

升级成：

需求输入 / 项目 → AI 分析 → AI 测试用例生成 → 人工确认 → 用例资产 → 自动化草稿 → 自动化资产 → 执行 → 结果 → 回链来源

这才是阶段 2 的真正目标。

二、为什么要修订旧版阶段 2 路线图

旧版阶段 2 的方向没错，但基于你现在的最新代码，它已经不够准确，原因主要有 4 个。

1）AI 链路在旧版里权重不够

现在你的系统里已经明确存在：

AI 模型配置页

Prompt 配置页

生成配置页

需求分析页

生成结果页

任务详情页

AI 助手页

这说明 AI 不是附属能力，而是业务主链的一部分。

2）2.1 的对象层范围原来偏窄

旧版更偏：

项目

测试用例

但现在应该把这些也纳入：

需求分析对象

生成任务对象

生成结果对象

3）AI 助手在旧版里角色不够清楚

现在要明确：

独立助手页不是重点

嵌入业务链的 AI 助手才是重点

4）生成配置已经出现，不能再把 AI 理解成“模型 + Prompt + 按钮”

现在 AI 生成底座至少包括：

模型配置

Prompt 模板

生成配置

这三者一起构成 AI 生成链路基础。

三、阶段 2 的正式拆分（v2）

我建议阶段 2 正式固定成 5 段：

2.1 测试设计对象层收口
2.2 AI 生成链路与 AI 助手收口
2.3 自动化草稿中心
2.4 草稿到目标资产落地
2.5 设计资产到执行资产追踪闭环

这 5 段是递进关系，而且每一段都可以单独交付。

四、阶段 2 的总主链路

这条链建议你以后作为整个平台的“核心业务故事”固定下来：

测试设计项目 / 需求输入

→ 需求分析
→ AI 生成测试用例
→ 人工确认 / 编辑
→ 测试用例资产
→ 自动化草稿
→ API / Web / App 资产
→ 执行
→ 结果
→ 回链项目 / 用例 / 草稿 / AI 来源

这条链是你平台后续最核心的差异化价值。

五、2.1 测试设计对象层收口
目标

状态

用例数量

自动化摘要位

作为需求输入与分析过程对象。

应承接：

分析状态

模型 / Prompt / 生成配置来源摘要

最近分析时间

生成任务入口

C. 生成任务对象

作为 AI 生成过程对象。

应承接：

任务状态

使用模型

使用 Prompt

使用生成配置

触发时间

结果数量

失败信息

作为 AI 生成产物的中间对象。

来源任务

六、2.2 AI 生成链路与 AI 助手收口
目标

把目前分散存在的 AI 相关能力，收成一条可理解、可配置、可追踪、可确认的业务链。

2.2 必须纳入的能力
A. 模型 API 配置

明确：

哪些模型源存在

哪些场景用哪个模型

哪些参数可调

哪些参数平台默认

B. Prompt 模板配置

明确：

需求分析模板

用例生成模板

AI 助手模板

模板与场景关系

C. 生成配置

明确：

不同生成模式

场景参数

结果格式要求

D. 需求输入 / 上传

支持：

文本输入

文件上传

需求摘要生成

项目关联

E. AI 分析

把需求转成结构化分析结果。

F. AI 生成测试用例

生成测试用例草案，不直接等于最终资产。

G. 人工确认 / 编辑

这是关键步骤，不能跳过。

修改

勾选/筛选

保存为正式测试用例

AI 助手不是只做独立页，而要嵌入到：

成功还是失败

谁确认保存的

十一、修正版阶段 2 的风险控制
2.1 风险

容易变成“修页面”，忘了收对象关系。

2.2 风险

容易让 AI 配置、生成页、助手页继续各自为政。

2.3 风险

容易把草稿系统散到各模块里。

2.4 风险

容易过早追求“智能生成很完美”，忽略保存、跳转、失败反馈。

2.5 风险

容易设计一堆复杂追踪字段，反而没有最小闭环。
