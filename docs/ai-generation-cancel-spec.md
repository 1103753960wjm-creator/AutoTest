# AI 生成任务取消语义规范

## 1. 文档目标

本文档只约束 AI 生成任务的取消语义，不展开任务恢复、自动评审列表或完整结果确认流。

本轮目标是把“取消生成”收成应用层协作式真取消，而不是前端假取消。

## 2. 当前问题

当前存在两类问题：

- 前端取消只是停本地状态，不调用真实后端取消接口
- 后端虽然能把任务写成 `cancelled`，但后台执行链没有协作式取消检查，后续仍可能继续写结果并把任务覆写成 `completed` 或 `failed`

## 3. 冻结语义

本轮采用“应用层协作式真取消”。

固定要求如下：

- 取消按钮必须调用真实后端取消接口
- 取消成功后不清空 `taskId`，不立刻丢失任务上下文
- 后台执行链在关键阶段必须再次检查取消状态
- 所有最终结果写库前必须再次检查取消状态
- 已取消任务不得被后续写回污染

## 4. 后端检查点

必须至少在以下阶段执行取消检查：

1. 生成开始前
2. 生成完成后、写 `generated_test_cases` 前
3. 自动评审开始前
4. 自动评审完成后、写最终 `review_feedback` 前
5. 改写开始前
6. 改写完成后、写最终 `final_test_cases` 前
7. 写 `completed` 与 `completed_at` 前

流式 callback 允许按时间间隔检查，不要求每个 chunk 查库。

## 5. 取消后的结果处理

- 取消前已写入的中间内容允许保留
- 取消后不得继续推进后续阶段
- 取消后不得继续写新的最终结果
- 取消后不得再写 `completed`
- 通用异常处理前必须先刷新任务状态；若任务已是 `cancelled`，不得再覆写成 `failed`

## 6. 前端行为

### RequirementAnalysisView

- 取消成功后：
  - 停止 tracker
  - 保留 `currentTaskId`
  - 再拉一次 `progress`
  - 进入“已取消，可继续查看任务”状态

- 取消失败后：
  - 不清空任务上下文
  - 不伪装成取消成功

### TaskDetail

- 运行中任务应允许取消
- `cancelled / failed / completed` 不显示取消按钮

## 7. 验收标准

- 点击取消会调用真实后端取消接口
- 任务最终状态稳定为 `cancelled`
- 不再出现前端显示已取消、后台却继续跑完并覆写状态的问题
- 取消后任务上下文仍可查看
