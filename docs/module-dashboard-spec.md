# 阶段 1 - 模块 Dashboard 统一规范

## 1. 目标

本轮仅统一一级模块首页的 Dashboard 骨架，不重写平台壳、不重写页面头部，也不改模块内部业务页。

第一轮强调：

- 同骨架，不同内容
- 结构统一优先于数据满配
- 优先复用 `StatCard`、`QuickActionCard`、`RecentList` 和状态组件
- 风险提醒宁可轻占位，也不硬造假数据

## 2. 第一批试点模块

本轮仅落地以下 3 个模块：

1. 接口自动化
2. Web 自动化
3. App 自动化

暂缓模块：

- 测试设计：当前更接近模块入口与列表群，若现在强做 Dashboard，容易和 1.8 列表页统一混在一起。
- 数据工厂：当前更像工具工作区，不适合在这一轮强行改造成 Dashboard。

## 3. 统一骨架

每个模块 Dashboard 统一为以下四段：

### 3.1 模块概览

- 使用 `StatCard`
- 控制在 3 到 4 张
- 展示模块级轻量概览，不堆复杂图表

### 3.2 常用入口 / 快捷动作

- 使用 `QuickActionCard`
- 强调“继续进入哪个高频工作区”
- 不把模块内部多个页面拼成第二套导航

### 3.3 最近动态

- 使用 `RecentList`
- 优先接入模块已有真实记录流或最近执行数据
- 支持刷新、空态、加载态和错误态

### 3.4 风险提醒 / 最近失败

- 优先接入真实失败记录或可稳定推导的数据
- 若当前没有稳定接口，允许保留占位结构
- 占位内容必须明确标记“后续接入执行中心 / 统一报告源”，不得硬拼不稳定逻辑

## 4. 页面头部边界

- 页面头部始终由 Layout 挂载的 `PlatformPageHeader` 承接
- 模块 Dashboard 通过 `route meta + usePlatformPageHeader` 注册页面级状态、更新时间、辅助说明和主操作
- 模块页面内部不得再自行渲染第二套页面头部

## 5. 第一批模块数据边界

### 5.1 接口自动化

真实数据：

- Dashboard 统计数据
- 操作日志

占位结构：

- 风险提醒 / 最近失败

说明：

- 当前没有稳定失败流接口，本轮只保留风险区块落位，不伪造失败数据。

### 5.2 Web 自动化

真实数据：

- Dashboard 统计数据
- 操作记录

占位结构：

- 风险提醒 / 最近失败

说明：

- 当前已有数据更偏操作记录，暂不从列表页或脚本页反向拼接失败流。

### 5.3 App 自动化

真实数据：

- Dashboard 统计数据
- 最近执行记录

可轻推导数据：

- 最近失败 / 执行异常，可从最近执行记录中筛出失败项

占位结构：

- 当最近失败为空时，用轻占位说明后续接执行中心和报告源

## 6. 不在本轮处理的内容

- 不重写 Layout
- 不新增第二套页面头部体系
- 不重做 Home
- 不重做最近访问 / 收藏 / 搜索
- 不大改模块内部列表页、编辑页、工作区
- 不提前进入 1.8 列表页统一
- 不为了“完整感”堆大量图表或运营看板

## 7. 后续接入建议

- 1.8 列表页统一时，只处理模块内部列表页主体结构，不回头改 Dashboard 骨架
- 执行中心或统一报告源成熟后，再逐步替换各模块的风险占位结构
- 后续新增模块 Dashboard 时，优先沿用本规范的四段式骨架
