# 统一页面壳定义

## 1. 目标

本次任务只做“页面壳定义”和“最小可复用骨架”，不做全站替换，不改业务接口，不改页面业务逻辑。  
目标是给 TestHub 后续页面收敛提供统一的结构语言，避免每个页面继续自行拼装头部、操作区、筛选区、内容区和结果区。

## 2. 现状判断

结合当前前端代码，页面结构大致分成 4 类：

1. `Dashboard` 类  
   典型问题：卡片区、快速操作区、图表区、活动区都是页面自己手写，缺少统一的头部和主次内容分区。
2. `List` 类  
   典型问题：列表页通常都有“标题 + 主操作 + 筛选栏 + 表格 + 分页 + 弹窗”，但每页都在重复搭骨架。
3. `Workspace` 类  
   典型问题：接口管理、复杂编辑器、数据工厂这类页面存在明显的多栏布局、工具栏和工作区，但没有统一壳层来表达“左侧资源 / 中间工作区 / 右侧辅助区”。
4. `Detail / Result` 类  
   典型问题：报告、详情、结果页既需要顶部摘要，也需要主结果区和辅助说明区，但当前结构不统一。

## 4. 页面壳定义

### 4.1 DashboardShell

解决的问题：

- 统一概览页的头部、指标区、主内容区、侧内容区。
- 让 dashboard 页面默认具备“标题 + 指标 + 主内容 + 次要内容”的清晰层次。

标准结构：

1. 页面头部：标题、描述、右侧动作
2. 指标区：统计卡片、关键 KPI
3. 概览区：快速操作、摘要信息、辅助说明
4. 内容区：主图表 / 主列表
5. 次内容区：活动流、说明卡、补充信息

推荐插槽：

- `header`
- `actions`
- `metrics`
- `overview`
- 默认插槽
- `secondary`
- `footer`

适用页面：

- 各子系统 dashboard
- 平台工作台
- 概览型报告首页

后续迁移建议：

- 先迁各模块 dashboard 页
- 先替换结构，不动图表和数据获取逻辑
- 旧页面中的统计卡片本身可保留，只把外层区块换成壳

### 4.2 ListShell

解决的问题：

- 统一资产管理页常见的“标题 + 新建按钮 + 筛选栏 + 表格 + 分页 + 弹窗挂载位”。
- 减少列表页对 `page-header`、`filter-bar`、`card-container` 的重复实现。

标准结构：

1. 页面头部：标题、描述、主操作按钮
2. 工具区：次级操作、批量操作、状态说明
3. 筛选区：搜索、状态筛选、时间筛选
4. 内容区：表格 / 卡片列表
5. 分页区：分页器、总数
6. 对话框区：创建、编辑、删除确认等附属弹窗

推荐插槽：

- `header`
- `actions`
- `toolbar`
- `filters`
- 默认插槽
- `pagination`
- `dialogs`
- `footer`

适用页面：

- 项目管理
- 用例管理
- 套件管理
- 环境管理
- 日志列表

后续迁移建议：

- 先迁结构最标准的后台列表页
- 表格列定义、分页逻辑、对话框逻辑不动
- 筛选栏和内容区按 slot 平移

### 4.3 WorkspaceShell

解决的问题：

- 统一复杂工作台页面的多栏布局。
- 把“左侧资源区 / 中间工作区 / 右侧辅助区”定义成可复用结构，而不是每个页面自己实现。

标准结构：

1. 页面头部：标题、描述、全局动作
2. 工作区主体：
   - 左侧资源区：树、列表、目录、导航
   - 中间主工作区：编辑器、画布、表单、操作台
   - 右侧辅助区：结果、详情、预览、日志
3. 工具栏：主工作区上方的次级操作区
4. 底部区：补充结果、说明、状态提示

推荐插槽：

- `header`
- `actions`
- `sidebar`
- `toolbar`
- 默认插槽
- `aside`
- `footer`

适用页面：

- API 接口管理
- Web 脚本编辑器
- App 用例编排
- 数据工厂复杂页

后续迁移建议：

- 先迁多栏最明显的工作台
- 左右栏原组件不拆，只替换外层布局
- 保持现有请求、保存、调试、执行逻辑不动

### 4.4 DetailResultShell

解决的问题：

- 统一详情、执行结果、测试报告页面的摘要区、主结果区和辅助信息区。
- 让结果页在视觉上更容易区分“摘要信息”和“详细内容”。

标准结构：

1. 页面头部：标题、描述、导出 / 返回 / 操作按钮
2. 摘要区：关键数字、状态卡、结果概览
3. 元信息区：筛选器、标签、描述、上下文信息
4. 主内容区：图表、日志、明细、详情主体
5. 辅助区：说明、侧边摘要、排行榜、附加信息

推荐插槽：

- `header`
- `actions`
- `summary`
- `meta`
- 默认插槽
- `aside`
- `footer`

适用页面：

- 测试报告
- 执行详情
- 任务结果
- 详情页

后续迁移建议：

- 先迁报告和详情页的外层结构
- 图表实例、日志明细、详情字段渲染逻辑不动
- 把页面顶部的筛选栏和导出按钮优先放到 `meta` / `actions`

## 6. 统一约束

1. 页面壳只负责结构，不负责请求、状态和业务逻辑。
2. 页面壳默认兼容当前 Element Plus 风格，不引入新 UI 体系。
3. 页面壳优先提供 slot，而不是把业务行为抽成复杂 props。
4. 页面壳的目标是降低后续迁移成本，不是一次性统一所有样式细节。

## 7. 当前阶段不做

- 不替换现有页面
- 不统一所有 class 命名
- 不重构现有 dashboard / list / workspace / result 页面逻辑
- 不改 layout、router、api、store
