# 规格说明

## 背景
- `M3 第一刀` 已经完成了最小主链：
  - 从 `approved` 测试用例生成 `api / web / app` 自动化草稿
  - 草稿先入 `Automation_draft`
  - 人工确认后保存到现有资产体系
- 但当前闭环仍然偏“能跑通”，离“能顺着用”还有明显缺口：
  - 保存后只能跳到目标模块入口，不能精确定位到新生成的资产
  - 测试用例资产页只有“生成”入口，没有直接展示已生成草稿或目标资产的追溯状态
  - 自动化草稿页虽然能展示历史列表，但不能直接点开某条历史草稿继续看
  - API / Web / App 现有编辑页还不支持读取路由参数后自动定位到目标资产
- 所以 `M3 第二刀` 不再解决“有没有”，而是解决“能不能顺着接着用”。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本文件只做需求对齐，不作为直接编码信号。
- 继续复用现有 API / Web / App 资产页面，不平行再做一套新编辑器。
- 所有新增文档、说明、代码注释统一使用中文。
- AI 生成逻辑仍然只能通过统一 `AI Gateway` 进入。

## 目标
- 保存自动化草稿后，前端能精确跳转并定位到刚创建的目标资产。
- 测试用例资产页可以直接看到该用例已有的自动化草稿和目标资产追溯摘要。
- 用户可以从测试用例资产页或自动化草稿页再次打开已有草稿，而不是只能重新生成。
- 让 `M3 第一刀` 形成真正可连续操作的工作流：
  - 生成
  - 查看
  - 保存
  - 回跳
  - 再次进入

## 非目标
- 本次不做批量转换。
- 本次不做自动执行。
- 本次不做草稿与目标资产的双向同步。
- 本次不做复杂版本树和差异对比。
- 本次不做 Web / App / API 目标页的大改版。

## 当前缺口

### 缺口 1：保存后跳转不精确
- 当前后端只返回模块级 `target_route`。
- 前端保存后只能进入目标模块首页，用户仍需自己再找刚生成的资产。

### 缺口 2：测试用例页没有追溯摘要
- 当前测试用例资产页只提供 `生成 API / Web / App 草稿` 按钮。
- 用户看不到：
  - 是否已有草稿
  - 是否已保存到目标资产
  - 对应目标资产编号

### 缺口 3：已有草稿不能顺手重开
- 当前自动化草稿页有历史列表，但没有直接“打开某条历史草稿”的动作。
- 测试用例资产页也没有“继续编辑最近草稿”的入口。

### 缺口 4：目标页不认路由参数
- 当前 API / Web / App 编辑页没有消费自动化草稿跳转参数。
- 即使后端返回了目标资产编号，前端也无法自动选中对应资产。

## 核心流程

### 流程 A：从测试用例页继续打开已有草稿
1. 用户进入测试用例资产页。
2. 页面显示该用例的自动化草稿摘要。
3. 用户点击：
  - `继续编辑 API 草稿`
  - `继续编辑 Web 草稿`
  - `继续编辑 App 草稿`
4. 页面进入自动化草稿确认页，并直接打开对应 `draft_id`。

### 流程 B：保存后精确回跳
1. 用户在自动化草稿页点击保存到目标资产。
2. 后端返回：
  - `target_route`
  - `target_route_query`
  - `target_asset_id`
  - `target_menu_id`
3. 前端跳转到目标页面。
4. 目标页面根据路由参数自动定位到刚生成的资产。

### 流程 C：从测试用例页查看追溯结果
1. 用户打开测试用例资产页。
2. 页面展示每条测试用例的自动化摘要。
3. 用户能直接看到：
  - 最新草稿状态
  - 最新保存目标类型
  - 目标资产编号
  - 快速入口

## 页面行为

### 测试用例资产页
- 每条用例新增“自动化进度”展示区。
- 对每个目标类型显示两类动作：
  - 如果还没有草稿：`生成草稿`
  - 如果已有草稿：`继续编辑`
- 如果已有保存结果，再增加：
  - `查看目标资产`

### 自动化草稿页
- 历史草稿列表增加：
  - `打开`
  - `查看目标资产`
- 当前页打开某条历史草稿时，不重新生成。
- 页面顶部明确展示：
  - 当前是新生成草稿还是历史草稿
  - 当前保存状态

### API 目标页
- 读取路由参数中的 `asset_id`
- 自动定位到对应 `Api_script`
- 第一刀后的第二刀要求至少做到：
  - 自动查询到目标资产
  - 自动选中或高亮该记录

### Web 目标页
- 读取 `menu_id`
- 自动选中对应菜单节点
- 自动打开对应脚本 tab 或展示对应脚本内容

### App 目标页
- 读取 `menu_id`
- 自动选中对应菜单节点
- 自动打开对应脚本页签

## 路由参数约定

### API
- `target_route = "/_api_script"`
- `target_route_query`
  - `asset_id`
  - `draft_id`
  - `source=testcase_automation`

### Web
- `target_route = "/web"`
- `target_route_query`
  - `menu_id`
  - `draft_id`
  - `source=testcase_automation`

### App
- `target_route = "/app_auto"`
- `target_route_query`
  - `menu_id`
  - `draft_id`
  - `source=testcase_automation`

## 关键约束
- 第二刀不能破坏第一刀已有的生成与保存主链。
- “继续编辑已有草稿”必须优先打开已有 `draft_id`，不能默认又生成一条新草稿。
- 测试用例资产页展示的是摘要，不在列表页直接展开完整草稿内容。
- 目标页对路由参数的支持，必须以“最小改动可定位”为原则，不强行推翻现有页面结构。
- 如果目标资产已被删除，前端需要给出明确提示，不能静默失败。

## 风险点
- 风险 1：如果在测试用例列表页展示过多自动化详情，会把资产页变得过重。
- 风险 2：如果目标页自动定位逻辑写得太深，容易和现有页面初始化流程冲突。
- 风险 3：如果“继续编辑”误走成“重新生成”，草稿数量会快速膨胀。
- 风险 4：如果只返回 `target_route` 不返回精确 query，第二刀目标就无法实现。

## 验收标准
- [ ] 测试用例资产页能展示自动化摘要
- [ ] 测试用例资产页能从已有草稿直接进入自动化草稿页
- [ ] 自动化草稿页能打开历史草稿，而不是只显示静态列表
- [ ] 保存成功后后端返回 `target_route_query`
- [ ] API 目标页可根据路由参数定位到新生成的 `Api_script`
- [ ] Web 目标页可根据路由参数定位到新生成的 `Web_menu / Web_script`
- [ ] App 目标页可根据路由参数定位到新生成的 `App_menu / App_script`
- [ ] 若目标资产被删除或不存在，页面提示明确
- [ ] 至少有后端 pytest、前端类型校验覆盖第二刀改动

## 失败场景
- [ ] 已有草稿存在，但“继续编辑”又生成了新草稿
- [ ] 保存成功后跳转到目标页，但页面没有定位到目标资产
- [ ] 测试用例资产页自动化摘要与真实草稿记录不一致
- [ ] 目标资产已删除，但“查看目标资产”无提示
- [ ] 路由参数缺失时，目标页进入异常状态
