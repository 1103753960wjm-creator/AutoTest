# 规格说明

## 背景
- 当前项目本质上还是“测试执行平台”，强项在 API、Web、App 自动化执行、调度、报告和通知。
- 你现在要做的不是一个小功能，而是一次平台级改造：把项目从“执行平台”升级成“测试生产平台”。
- 本次需求已经明确：
  - 整个平台必须支持三种模式切换：`none`、`local_llm`、`remote_llm`
  - 平台要能根据需求文档或功能设计输入自动生成测试用例
  - 生成出来的内容必须先让人审，再进入后续资产管理或自动化草稿生成
  - 工程规范和全局记忆也要同步支持这次改造

## 产品目标
- 把 L-Tester 从“执行测试的平台”升级成“生产测试资产的平台”。
- 在保住现有 API、Web、App 执行能力的前提下，补齐“需求 -> 测试用例 -> 自动化草稿”的上游链路。
- 所有 AI 能力必须经过一个统一网关治理，不能散落在业务模块里。

## 核心需求

### R1. 三种运行模式
- `none`
  - 不依赖本地模型，也不依赖远程模型。
  - 平台仍然要能基于规则和模板生成测试用例草稿。
- `local_llm`
  - 平台通过统一入口调用本地部署的大模型。
  - 典型目标包括 Ollama、LM Studio、vLLM 等 OpenAI 兼容服务。
- `remote_llm`
  - 平台通过统一入口调用远程大模型服务。
  - 典型目标包括 OpenAI 兼容的云端服务。

### R2. 基于需求生成测试用例
- 用户可以输入需求文本、功能设计文本，后续再扩展到文档上传。
- 平台需要把输入解析成结构化内容，再生成测试用例草稿。
- 生成结果不是最终执行脚本，而是可审、可改、可追溯的测试用例资产草稿。

### R3. 人工审核必不可少
- AI 生成的测试用例草稿必须经过人工确认，才能成为正式测试用例资产或继续转换成自动化草稿。
- 平台必须保留需求来源、生成结果、后续脚本草稿之间的可追溯关系。

### R4. 统一 AI 治理
- 所有 AI 相关能力必须走统一的 AI Gateway。
- API、Web、App、调度、通知、报告等模块不得直接调用模型。
- provider 配置、prompt 模板、回退策略、超时、失败处理等都要统一治理。

## 非目标
- 本次改造不替换现有 API/Web/App 执行模块。
- 本次改造不允许需求输入绕过人工审核后直接进入生产执行。
- 第一阶段不要求模型微调。
- 第一阶段不要求文档 RAG、向量库、知识库。
- 第一阶段不要求“一键生成最终自动化脚本并立即执行”。

## 当前假设
- 第一阶段采用“平台级全局模式切换”，不是“按项目切换”或“按任务切换”。
- 第一阶段的测试用例生成输入只支持纯文本；文件上传和文档解析后续再做。
- 第一阶段审核流允许采用简单的“草稿 -> 已审核”模式，复杂审批流后续再补。
- 改造过程中必须保证现有 API/Web/App 执行链路行为稳定，不因新功能回归。
- 第一阶段前端菜单落点明确为：
  - AI 模式配置进入 `system` 体系
  - 测试用例生成进入一级业务菜单，不放进 `system`

## 目标产品流程

### 流程 A：模式切换
1. 管理员进入 `system` 下的 AI 配置页面。
2. 平台展示当前模式：`none`、`local_llm`、`remote_llm`。
3. 管理员切换模式，并根据模式配置 provider 信息。
4. AI Gateway 对外暴露统一的生效模式。

### 流程 B：从需求输入生成测试用例
1. 用户从一级业务菜单进入测试用例生成页面，创建需求项或直接粘贴功能设计文本。
2. 平台将输入归一化为可处理的需求内容。
3. AI Gateway 在当前模式下生成测试用例草稿。
4. 用户对测试用例草稿进行查看、编辑、接受或拒绝。
5. 被接受的测试用例草稿保存为正式测试用例资产，并保留来源追溯信息。

### 流程 C：从测试用例生成自动化草稿
1. 用户选中一个已经审核通过的测试用例。
2. 平台为 API、Web 或 App 生成自动化草稿。
3. 人工继续修正或补全自动化草稿。
4. 执行模块运行被确认后的自动化资产。

## 目标架构

### 1. AI Gateway
- 后端模块：`l-tester-master/views/ai`
- 前端配置入口：`l-vue-ui-master/src/views/system/*`
- 职责：
  - 模式解析
  - provider 选择
  - prompt 分发
  - 回退处理
  - AI 输出结构统一
- 强约束：
  - API、Web、App 业务模块不得直接接模型

### 2. 需求中心
- 后端模块：`l-tester-master/views/requirement`
- 前端模块：`l-vue-ui-master/src/views/requirement_view`
- 职责：
  - 需求输入接收
  - 需求标准化存储
  - 来源版本管理
  - 与测试用例生成任务建立追溯关系

### 3. 测试用例中心
- 后端模块：`l-tester-master/views/testcase`
- 前端模块：`l-vue-ui-master/src/views/testcase_view`
- 菜单层级：一级业务菜单
- 职责：
  - 测试用例草稿生成
  - 测试用例编辑与审核
  - 测试用例版本管理
  - 与自动化草稿之间建立关联

### 4. 自动化草稿适配层
- 职责：
  - 把审核通过的测试用例转换成 API、Web、App 自动化草稿
  - 输出必须可编辑，不能强制全自动执行

### 5. 治理层
- 职责：
  - provider 配置
  - prompt 模板配置
  - 生成日志
  - 超时与重试策略
  - 成本和质量度量

- 目标：

  - 前端报告地址配置收口

### M1. AI 基础层
- 目标：
  - 一个 AI Gateway
  - 一个平台级模式切换
  - 一个基于纯文本的测试用例草稿生成闭环
- 包含：
  - 后端 AI Gateway

  - 执行前人工修正能力

### M4. 治理与质量闭环

  - 让 AI 能力可治理、可审计、可度量

  - 调用统计
  - 失败分析
  - 质量评分

### M1 范围
- 后端：
  - AI Gateway
  - provider 契约

  - 测试用例生成接口

  - `system` 下的最小 AI 模式配置页

  - 仅支持纯文本输入

  - 暂不支持直接转换成自动化脚本

### M1 验收标准
- [ ] 平台可以展示并切换 `none`、`local_llm`、`remote_llm`
- [ ] `none` 模式无需模型即可生成结构化测试用例草稿
- [ ] `local_llm` 和 `remote_llm` 共享统一网关契约
- [ ] 用户可以粘贴需求文本或功能设计文本并获得测试用例草稿
- [ ] 用户可以在保存前手动编辑或丢弃测试用例草稿
- [ ] AI 相关配置没有散落硬编码在业务页面或业务模块中
- [ ] 至少有后端 pytest 和一个前端构建或类型校验覆盖本阶段改动

## 失败场景

### 模式失败
- [ ] 配置模式非法
- [ ] 目标 provider 未配置
- [ ] provider 地址存在但不可达
- [ ] 前端展示模式与后端生效模式不一致

### 生成失败
- [ ] 需求输入为空或过于模糊
- [ ] 测试用例草稿缺少关键字段
- [ ] 三种模式的输出结构不一致
- [ ] 大模型返回非法 JSON

### 治理失败
- [ ] 业务模块绕过 AI Gateway 直接调用模型
- [ ] provider 配置被写死在源码中
- [ ] prompt 和生成日志无法追溯

### 回归失败
- [ ] 现有 API、Web、App 执行链路受影响
- [ ] 调度、通知、报告链路因为改造发生回归
- [ ] 前端菜单或权限行为因为新页面接入发生回归

## 优化方向

### 产品方向
- 增加“需求 -> 测试用例”的审核闭环，而不是只做执行闭环。
- 增加测试用例质量评分和重复检测。
- 增加测试用例版本对比。

### 架构方向
- 所有 AI 能力统一收口到网关。
- 把需求资产、测试用例资产、自动化资产明确分层。
- 通过适配层连接测试用例和执行模块，避免直接耦合。

### 工程方向
- 继续清理硬编码密钥和路径。
- 补齐 AI Gateway、需求解析、草稿转换等关键路径的 pytest。
- 统一前端环境配置和后端运行配置。

### 治理方向
- 增加 prompt 模板版本管理。
- 增加生成日志和审计日志。
- 增加超时、重试、回退和 provider 健康状态可视化。

### 质量方向
- 增加测试用例完整度基准。
- 增加“必须人工审核”的防护。
- 为鉴权、调度、报告、设备控制等高风险链路补回归验证。
