# 规格说明

## 背景
- `002-platform-ai-rebuild.md` 中已经把平台改造拆成 `M0 -> M4`。
- 当前 `M1` 已经具备：
  - AI Gateway
  - 平台级模式切换
  - 纯文本测试用例草稿生成
  - 前端人工编辑、标记审核、丢弃草稿
- 当前缺口是：测试用例草稿还只是页面内临时状态，刷新后即丢失，无法成为平台资产。

## 本次切片定位
- 本次实现属于 `M2. 需求与测试用例持久化` 的第一刀。
- 本次切片只解决“已审核草稿如何成为平台资产”。
- 本次切片不进入 `M3. 自动化草稿生成`。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 继续遵守 AI 能力只能通过统一入口接入，不允许业务模块直连模型。
- 新增或修改文档、代码注释统一使用中文。
- 后端继续保持 `view -> common -> model` 的分层方式。
- 前端继续保持 `view -> api -> axios -> backend` 的调用链路。

## 目标
- 把“已审核测试用例草稿”保存为：
  - 需求资产
  - 测试用例资产
  - 生成任务记录
- 用户在“用例生成”页可以查看最近保存的需求资产。
- 用户可以按需求查看已保存的测试用例明细。

## 范围

### 包含
- 后端新增 `requirement` 模块：
  - requirement 模型
  - generation job 模型
  - 保存已审核草稿接口
  - 最近需求资产列表接口
- 后端新增 `testcase` 模块：
  - testcase 模型
  - 按需求查询测试用例接口
- 前端更新“用例生成”页：
  - 增加“保存已审核草稿”按钮
  - 增加“最近需求资产”列表
  - 增加“查看已保存用例”抽屉
- 新增最小 pytest：
  - 保存逻辑
  - 审核过滤逻辑
  - 追溯关系校验

### 不包含
- 自动化草稿生成
- requirement/testcase 独立管理页面
- 复杂审批流
- 版本对比
- 文件上传与文档解析

## 接口草案

### 1. 保存已审核草稿
- 路径：`POST /api/requirement/save_reviewed_cases`
- 输入：
  - 需求标题
  - 来源类型
  - 原始需求内容
  - 需求摘要
  - 当前模式信息
  - 已审核测试用例列表
- 输出：
  - requirement 信息
  - generation job 信息
  - 保存成功的 testcase 数量

### 2. 最近需求资产列表
- 路径：`POST /api/requirement/list_recent`
- 输入：
  - 用户信息
  - 可选数量限制
- 输出：
  - 最近需求资产列表
  - 每条资产的 testcase 数量
  - 最近一次生成模式信息

### 3. 按需求查询测试用例
- 路径：`POST /api/testcase/list_by_requirement`
- 输入：
  - `requirement_id`
- 输出：
  - requirement 基本信息
  - 对应 testcase 列表

## 前端交互
- 用户在“用例生成”页生成并人工审核测试用例。
- 只有被标记为“已审核”的测试用例允许保存。
- 保存成功后，页面刷新“最近需求资产”列表。
- 用户可点击“查看用例”查看该需求下的测试用例明细。

## 验收标准
- [ ] 已审核的测试用例草稿可以保存为 requirement、generation job、testcase 三类实体
- [ ] 未审核的测试用例不会被保存
- [ ] 最近需求资产列表能展示标题、来源类型、保存时间、用例数量、模式信息
- [ ] 用户可按需求查看已保存 testcase 明细
- [ ] 保存逻辑至少有 pytest 覆盖
- [ ] 不影响现有 AI Gateway 和草稿生成逻辑

## 失败场景
- [ ] 未传需求标题或需求内容
- [ ] 没有任何已审核测试用例
- [ ] 保存成功 requirement，但 testcase 保存不完整
- [ ] requirement 与 generation job 追溯关系不一致
- [ ] 前端把未审核草稿误保存

## 回退策略
- 如果本切片被否决，只需要移除：
  - `views/requirement`
  - `views/testcase`
  - 前端保存与资产查看 UI
- 不得影响现有 AI 模式配置和测试用例草稿生成链路。
