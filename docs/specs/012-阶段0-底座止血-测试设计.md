# 测试设计

## 背景
- 本文对应 [012-阶段0-底座止血.md](/e:/AutoTest/docs/specs/012-阶段0-底座止血.md)。
- 本轮目标不是扩功能，而是验证底层基础链路不会在止血时回归。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本轮优先覆盖登录、菜单、路由、基础请求链路。
- 第一阶段之外的主控台改版、巨石页面拆分不在本轮验证范围内。

## 测试目标
- 证明存储工具 JSON 读写修复后不再触发运行时错误。
- 证明权限 getter 修复后仍能拿到按钮列表。
- 证明请求链路收敛后，现有页面接口不回归。
- 证明动态路由初始化不回归。
- 证明用例生成、草稿保存、资产定位相关主链仍可继续工作。

## 测试分层

### 一、前端静态校验
- `npm.cmd run type:check`
- 必须通过

### 二、前端最小调用级验证
- 存储工具：
  - `putJSON/getJSON` 正常写读
  - JSON 解析失败时有容错
- auth store：
  - `getButtonList` 能返回 `buttons`
- axios：
  - 请求拦截器不再假设 `config.data` 一定存在
  - token/header 注入逻辑不破坏 post/get/export/upload

### 三、路由与菜单验证
- 动态路由初始化后：
  - 菜单仍能生成
  - 空菜单时仍会回登录
  - 不再依赖调试输出判断状态

### 四、回归验证
- 测试用例列表页可正常查询
- 自动化草稿页可正常打开
- 自动化资产定位跳转不回归
- 主内容区滚动和测试用例页可读性不回归

## 失败场景

### 失败场景 1：token/header 收敛后接口全部失效
- 预期：
  - 先在最小联调链路暴露
  - 不允许上线后靠用户反馈发现

### 失败场景 2：导出接口因 header 改动失效
- 预期：
  - 导出链路单独验证
  - 不允许只验证普通 post

### 失败场景 3：动态路由无菜单或菜单解析失败
- 预期：
  - 明确回登录或给出错误反馈
  - 不允许静默空白页

### 失败场景 4：存储 JSON 容错缺失导致首页或用户态异常
- 预期：
  - 旧脏数据存在时不直接炸页面

## 最小验收矩阵
- [ ] `storage.ts` 修复后，JSON 读写通过
- [ ] `auth.ts` getter 修复后，按钮列表可读取
- [ ] `axios.ts` 收敛后，普通接口与导出接口都可用
- [ ] 动态路由初始化不回归
- [ ] 前端 `typecheck` 通过
- [ ] 至少一条业务主链人工或自动验证通过
