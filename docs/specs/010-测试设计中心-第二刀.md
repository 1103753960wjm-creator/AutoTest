# 规格说明

## 背景
- [009-测试设计中心-第一刀.md](/e:/AutoTest/docs/specs/009-测试设计中心-第一刀.md) 已经完成了第一阶段闭环：
  - 正式测试用例可编辑
  - 审核意见与审核记录
  - 批量基础修改
  - 版本快照基础
  - 正式测试用例页主表格重排
  - 中文映射与 Excel 导出
- 但从“测试设计效率”视角看，当前页面仍然偏第一版资产治理页，还不是真正顺手的设计中心。
- 目前仍缺少的高频设计能力主要有：
  - 版本之间的直接对比
  - 可筛选、可批量处理的标签体系
  - 更结构化的审核原因
  - 面向设计管理的批量操作增强
- 如果第二刀不补，正式测试用例虽然“能改”，但在多版本维护、多类型筛选、批量复核这些高频动作上仍然偏笨重。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`，本文件只做需求对齐，不作为直接开工信号。
- 本阶段仍然优先“测试设计中心”，不回跳 `008` 的 AI 治理实现。
- 所有新增文档、说明、代码注释统一使用中文。
- 继续复用现有 `testcase` 模块、测试用例资产页和现有自动化草稿主链，不平行再造一套页面体系。
- 本阶段不得破坏：
  - 正式测试用例编辑
  - 审核记录
  - 自动化草稿入口
  - 自动化追溯摘要
  - Excel 导出

## 阶段定位
- 本阶段是“测试设计中心第二刀”。
- 优先级继续高于 [008-m4-治理与质量闭环-第一刀.md](/e:/AutoTest/docs/specs/008-m4-治理与质量闭环-第一刀.md)。
- 当前设计中心路线调整为：
  1. 第一刀：正式测试用例可编辑、可审核、可批量改、可导出
  2. 第二刀：版本对比、标签体系、结构化审核原因、批量增强
  3. 第三刀：前后置依赖、场景链路、全链路追踪视图
  4. 之后再进入 AI 治理与质量闭环

## 目标
- 让正式测试用例不仅“能改”，还要“容易筛、容易比、容易批量处理”。
- 让审核记录从纯文本备注，升级成可统计、可筛选的结构化原因。
- 让版本历史从“看列表”升级成“看差异”。
- 让测试设计人员能用标签组织资产，而不是只能靠模块名和标题硬搜。
- 保持当前主表格可读性，不再把治理字段重新堆回核心列区。

## 非目标
- 本次不做前后置依赖。
- 本次不做场景链路编排。
- 本次不做 `需求 -> 用例 -> 自动化 -> 任务 -> 报告` 的完整追踪大视图。
- 本次不做复杂富文本 diff 编辑器。
- 本次不做标签治理后台（如颜色、分组、排序后台）。
- 本次不做自动化草稿与正式测试用例的双向同步。
- 本次不做 AI prompt 管理、AI 日志治理、模型成本统计。

## 当前缺口

### 缺口 1：版本历史能看列表，不能看差异
- 当前已经有版本快照，但只能看到“有过哪些版本”，不能直接看两版差别。
- 这会导致：
  - 设计评审时仍然需要人工肉眼比对
  - 修改影响范围不清楚

### 缺口 2：审核记录有文本，没有结构化原因
- 当前审核意见能记录，但缺少结构化原因类型。
- 这会导致：
  - 无法快速统计“最常见驳回原因”
  - 无法按审核问题类型筛选
  - 批量复核时缺少统一标准

### 缺口 3：没有标签体系
- 当前只能按模块、标题、需求、审核状态、目标类型筛选。
- 这会导致：
  - “冒烟”“回归”“核心链路”“支付风险”“登录场景”这类横切维度无法表达
  - 批量复核与后续自动化选择缺少高效入口

### 缺口 4：批量操作仍偏基础
- 当前批量能力只覆盖基础字段变更。
- 对测试设计中心来说，第二刀至少要补：
  - 批量审核
  - 批量打标
  - 批量移除标签

## 核心流程

### 流程 A：版本对比
1. 用户进入测试用例资产页。
2. 用户打开某条用例的“历史”。
3. 用户选择两个版本进行对比。
4. 页面展示字段级差异，至少覆盖：
  - 所属模块
  - 用例标题
  - 前置条件
  - 测试步骤
  - 预期结果
  - 优先级
  - 目标类型
  - 是否可自动化
5. 第一刀要求“字段级差异结果”，不要求复杂逐字高亮编辑器。

### 流程 B：结构化审核
1. 用户发起审核动作。
2. 若通过，可选填写审核意见并选择原因类型。
3. 若驳回，必须填写驳回说明，并选择驳回原因类型。
4. 平台同时记录：
  - 审核状态
  - 审核意见
  - 原因类型
  - 审核时间
5. 列表页支持按原因类型筛选。

### 流程 C：标签管理
1. 用户在测试用例编辑页维护单条标签。
2. 用户在列表页可直接查看标签摘要。
3. 用户可按标签筛选测试用例。
4. 第二刀允许：
  - 复用已有标签
  - 新建简单标签
  - 删除标签关联

### 流程 D：批量增强
1. 用户在测试用例资产页多选记录。
2. 用户可发起批量动作：
  - 批量审核
  - 批量打标
  - 批量移除标签
3. 平台返回成功数、失败数和失败原因摘要。

## 验收标准
- 正式测试用例可按标签筛选。
- 单条测试用例可新增、删除标签。
- 批量打标与批量移除标签可用。
- 审核时可以记录原因类型。
- 列表页可以展示最新审核原因类型的中文文案。
- 历史抽屉可以选择两个版本并返回字段级差异。
- 第一刀已实现的能力不得回归：
  - 编辑
  - 审核意见
  - 版本快照
  - 批量基础修改
  - 中文主表格
  - Excel 导出
  - 自动化草稿入口与风险提示

## 风险与约束
- 标签体系如果做成复杂后台，会拖慢第二刀，不符合当前节奏；本轮只做轻量标签。
- 版本对比如果直接做复杂文本 diff，前端复杂度会陡增；第二刀先做字段级 compare。
- 结构化审核原因必须控制枚举数量，不能一开始做成难维护的大字典。
- 本轮不能把 `用例类型 / 目标类型 / 审核状态 / 标签` 重新塞回主表格核心列区，避免第一刀刚修好的可读性回退。
