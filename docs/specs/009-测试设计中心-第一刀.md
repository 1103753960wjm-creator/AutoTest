# 规格说明

## 背景
- 当前平台已经具备：
  - 需求草稿生成
  - 测试用例草稿生成
  - 测试用例审核入库
  - 测试用例资产列表
  - 自动化草稿生成与落目标资产
- 但从“测试设计工作”视角看，现阶段仍更像“资产保存中心”，还不是“测试设计中心”。
- 当前正式测试用例的主要能力仍停留在：
  - 存下来
  - 列出来
  - 改审核状态
  - 继续生成自动化草稿
- 高频设计动作明显缺失：
  - 不能正式编辑已有测试用例内容
  - 没有审核意见、驳回原因、审核记录
  - 没有批量修改
  - 没有版本快照基础
  - 生成后草稿表格缺少统一表头约束与导出能力
- 如果这一层不补，测试设计仍然只能在“生成前草稿阶段”高效，正式资产一旦入库就变笨重。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`，本文件只做需求对齐，不作为直接开工信号。
- 本阶段优先解决“正式测试用例的设计效率”，暂不进入 AI 治理实现。
- 所有新增文档、说明、代码注释统一使用中文。
- 优先复用现有 `testcase` 模块、现有资产页和现有返回结构，不平行造一套新的设计系统。
- 本阶段不能破坏 `M3` 已打通的自动化草稿主链。

## 阶段定位
- 本阶段插入在 `M3` 之后、`M4` 之前。
- [008-m4-治理与质量闭环-第一刀.md](/e:/AutoTest/docs/specs/008-m4-治理与质量闭环-第一刀.md) 暂缓，不作为当前下一步。
- 当前优先级改为：
  1. 让正式测试用例可编辑、可审核、可批量处理
  2. 再补版本对比、标签、依赖、全链路追踪
  3. 最后进入 AI 治理与质量闭环

## 目标
- 让正式测试用例成为可持续编辑的一等设计资产，而不是一次性保存结果。
- 补齐单条编辑、审核意见、审核记录、批量基础修改。
- 为后续版本对比、标签体系、依赖关系、全链路追踪打基础。
- 在不打烂现有页面结构的前提下，把当前“测试用例资产页”升级成第一版“测试设计中心”。
- 统一“生成后的正式测试用例页”的主表格 schema，并补上 Excel 导出能力。

## 非目标
- 本次不做标签体系。
- 本次不做前后置依赖和场景链路编排。
- 本次不做 `需求 -> 用例 -> 任务 -> 报告` 完整追踪视图。
- 本次不做复杂版本对比界面。
- 本次不做批量导入导出。
- 本次不做自动化资产反向同步。
- 本次不做“编辑后自动回写已有自动化草稿或目标资产”。

## 当前缺口

### 缺口 1：正式测试用例不可编辑
- 当前正式测试用例一旦入库，接口层没有“编辑已有测试用例内容”的能力。
- 这会导致：
  - 修改标题、模块、步骤、预期结果只能回源重生或手工改库
  - 正式资产不能持续迭代

### 缺口 2：审核只有状态，没有意见
- 当前只有 `approved / rejected / draft` 状态，没有审核意见、驳回原因、审核动作记录。
- 这会导致：
  - 无法回答“为什么被驳回”
  - 无法回答“谁在什么时候改成什么状态”

### 缺口 3：没有批量基础操作
- 当前正式测试用例列表没有批量能力。
- 高频动作如批量改优先级、批量改目标类型、批量审核状态，都需要逐条处理。

### 缺口 4：没有版本基础
- 当前 `Testcase.version` 只有数字，没有真正的版本快照。
- 一旦正式编辑开启，如果没有快照，追溯会立刻断掉。

### 缺口 5：生成后的正式测试用例页展示不够标准化
- 当前测试用例资产页的主表格字段顺序和设计视角不一致，更偏“资产管理”而不是“测试设计”。
- 当前界面直接展示了内部英文编码，例如：
  - `happy_path`
  - `negative_path`
  - `permission_or_visibility`
  - `general`
- 第一刀需要把“生成后的正式测试用例页”主表格输出定成统一标准，并明确：
  - 主表格字段顺序
  - Excel 导出顺序
  - 英文编码只允许留在内部存储，不允许直接暴露在主界面

## 核心流程

### 流程 A：编辑正式测试用例
1. 用户进入测试用例资产页。
2. 用户点击某条用例的“编辑”。
3. 页面打开编辑抽屉或编辑面板。
4. 用户可以修改：
  - 所属模块
  - 用例标题
  - 前置条件
  - 测试步骤
  - 预期结果
  - 优先级
  - 目标类型
  - 是否可自动化
5. 用户保存后：
  - 正式测试用例内容更新
  - `version + 1`
  - 写入版本快照

### 流程 B：审核与驳回
1. 用户在列表页或编辑页发起审核动作。
2. 如果是驳回，必须填写驳回原因。
3. 平台写入：
  - 当前审核状态
  - 最新审核意见摘要
  - 审核记录
4. 列表页可直接看到最新审核结论。

### 流程 C：批量基础修改
1. 用户在测试用例资产页多选若干条记录。
2. 用户发起批量操作。
3. 第一刀只支持批量修改基础字段：
  - 优先级
  - 目标类型
  - 审核状态
  - 所属模块
4. 平台返回成功数、失败数和失败原因摘要。

### 流程 D：查看版本与审核记录
1. 用户在测试用例资产页打开某条用例的“历史”。
2. 页面可以查看：
  - 版本列表
  - 审核记录列表
3. 第一刀只要求可查看，不要求可视化 diff 或回滚。

### 流程 E：导出正式测试用例
1. 用户进入生成后的正式测试用例页。
2. 页面主表格按统一表头展示正式测试用例。
3. 用户点击“导出 Excel”。
4. 平台导出当前表格数据，列顺序与页面展示一致。

## 页面落点

### 测试用例资产页
- 保留现有筛选、自动化摘要和审核入口。
- 本阶段新增：
  - 单条编辑入口
  - 单条历史入口
  - 批量选择与批量操作栏
  - 最新审核意见简要展示
  - 主表格字段顺序重排
  - Excel 导出按钮

### 测试用例资产页主表格
- 主表格表头固定为：
  - `序号`
  - `所属模块`
  - `用例标题`
  - `前置条件`
  - `测试步骤`
  - `预期结果`
  - `优先级`
- `用例类型`、`目标类型`、`所属需求`、`自动化进度`、`审核状态` 不再占用主表格核心列区。
- 这些信息应转为：
  - 次级信息
  - 详情抽屉
  - 扩展面板
  - 操作区附属信息
- 主表格必须优先服务“测试设计阅读与编辑”，而不是资产治理字段堆叠。

### 中文展示规则
- 内部字段名和编码允许继续使用英文，例如：
  - `category`
  - `target_type`
  - `happy_path`
  - `general`
- 但页面主界面必须展示中文业务文案，不允许直接显示内部英文编码。
- 第一刀至少要求以下映射：
  - `happy_path` -> `主流程`
  - `negative_path` -> `异常流程`
  - `required_field_validation` -> `必填项校验`
  - `permission_or_visibility` -> `权限/可见性校验`
  - `general` -> `通用`
  - `api` -> `接口`
  - `web` -> `Web`
  - `app` -> `App`

### 编辑面板
- 采用抽屉或侧边编辑面板，不新开独立复杂页面。
- 原因：
  - 先复用当前资产页
  - 先让高频编辑动作落地
  - 不在第一刀就做重型设计工作台

### 历史面板
- 展示：
  - 版本列表
  - 审核记录列表
- 第一刀不做差异对比高亮，只展示摘要和时间线。

## 自动化草稿联动约束
- 如果测试用例已存在自动化草稿或目标资产，编辑正式测试用例后：
  - 不自动回写已有自动化草稿
  - 不自动改写 API / Web / App 目标资产
- 第一刀至少需要：
  - 明确提示“测试用例已变更，已有自动化草稿可能过期”
  - 或在列表中展示待人工确认的同步风险提示

## 关键约束
- 正式测试用例编辑必须保留 requirement 关联，不允许脱链。
- 驳回动作必须要求填写驳回原因，不能只有状态切换。
- 批量操作不允许直接批量改多行长文本字段。
- 每次正式内容变更前，都必须先写一份版本快照。
- 第一刀只做“版本可查看”，不做版本对比和回滚。
- 不允许因为测试设计中心改造而破坏当前自动化草稿入口和追溯摘要。
- 正式测试用例页的导出列顺序必须和页面表头顺序一致，不允许导出列与界面列脱节。
- 页面展示层必须做中文映射，不能把内部英文编码直接暴露给用户。

## 风险点
- 风险 1：如果第一刀就把设计页做成重型工作台，范围会立刻失控。
- 风险 2：如果没有版本快照就开放正式编辑，历史将不可追溯。
- 风险 3：如果批量操作范围过宽，容易误伤正式资产。
- 风险 4：如果编辑后不提示自动化资产可能过期，用户会误以为脚本仍然同步。
- 风险 5：如果正式测试用例页继续直接显示英文编码，用户会误以为字段未完成产品化。
- 风险 6：如果主表格继续塞入过多治理字段，测试设计阅读效率会持续很差。

## 验收标准
- [ ] 用户可以编辑已保存的正式测试用例内容
- [ ] 编辑保存后版本号递增
- [ ] 每次内容修改都会产生版本快照
- [ ] 用户可以提交审核意见或驳回原因
- [ ] 用户可以查看审核记录
- [ ] 用户可以批量修改优先级、目标类型、所属模块、审核状态
- [ ] 测试用例资产页能展示最新审核意见摘要
- [ ] 已有关联自动化草稿的测试用例在修改后会出现明确风险提示
- [ ] 生成后的正式测试用例页主表格表头固定为 `序号、所属模块、用例标题、前置条件、测试步骤、预期结果、优先级`
- [ ] 生成后的正式测试用例页支持导出 Excel
- [ ] 主界面不再直接显示 `happy_path`、`negative_path`、`general` 这类英文编码
- [ ] 至少有后端 pytest 和一个前端类型校验覆盖本阶段改动

## 失败场景
- [ ] 正式测试用例内容更新了，但 `version` 没有递增
- [ ] 驳回成功了，但没有留下驳回原因
- [ ] 批量修改部分失败时，没有返回失败项
- [ ] 审核记录存在，但列表页看不到最新审核意见
- [ ] 编辑正式测试用例后，没有留下版本快照
- [ ] 已生成自动化草稿的测试用例被修改，但没有任何风险提示
- [ ] 导出的 Excel 表头顺序与页面表头不一致
- [ ] 页面主表格仍直接显示英文编码而不是中文业务文案

- 暂不做：
