# 测试设计

## 背景
- 本文对应 [010-测试设计中心-第二刀.md](/e:/AutoTest/docs/specs/010-测试设计中心-第二刀.md)。
- 当前阶段只进入 `TDD`，不进入实现。
- 本文目标是提前锁定：
  - 第二刀需要证明哪些设计行为成立
  - 失败时应该怎样暴露
  - 最小回归面应该覆盖到哪里

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本轮只覆盖“测试设计中心第二刀”，不回跳 `008` 的 AI 治理实现。
- 所有新增文档、说明、代码注释统一使用中文。
- 第一刀已经修好的主表格可读性、页面滚动、正式编辑、审核、导出、自动化草稿入口不得回归。
- 本轮允许继续增强测试用例资产页，但不允许把治理字段重新挤回主表格核心阅读区。

## 本轮测试目标
- 证明历史抽屉支持两版字段级差异对比，而不只是版本列表。
- 证明审核动作支持结构化原因类型，并能在列表结果中体现。
- 证明测试用例支持标签维护、标签筛选和标签摘要展示。
- 证明批量打标、批量移除标签、批量审核增强可用。
- 证明第二刀新增能力不会破坏第一刀的正式编辑、审核记录、批量基础修改、Excel 导出和自动化草稿入口。

## 非目标
- 不验证前后置依赖。
- 不验证场景链路编排。
- 不验证 `需求 -> 用例 -> 自动化 -> 任务 -> 报告` 的完整追踪视图。
- 不验证复杂逐字高亮 diff 编辑器。
- 不验证 AI prompt 治理、AI 调用日志和模型成本统计。
- 不验证自动化资产双向同步。

## 测试分层

### 一、后端 pytest
用于验证：
- 版本对比结果
- 结构化审核原因
- 标签模型与关联
- 批量标签和批量审核
- 列表结果增强

优先新增或扩展的测试文件：
- `l-tester-master/tests/test_testcase_common.py`
- `l-tester-master/tests/test_testcase_view.py`

### 二、前端类型校验
用于保证：
- 列表页、编辑抽屉、历史抽屉新增字段后类型不破
- 标签筛选、标签编辑、对比结果渲染没有静态错误
- 第二刀不破坏第一刀已有导出和自动化入口

本轮最低要求：
- `npm.cmd run type:check`

### 三、前端交互核对清单
由于当前前端没有成熟组件单测基线，本轮关键交互先采用实现后核对清单：
- 标签筛选
- 标签摘要展示
- 标签编辑
- 批量打标 / 移除标签
- 审核原因类型选择
- 两版对比结果展示

### 四、回归检查
本轮必须显式回归：
- 正式测试用例单条编辑
- 审核意见与审核记录
- 批量基础修改
- 主表格 7 列
- 中文显示
- Excel 导出
- 自动化草稿入口与风险提示
- 页面可滚动、主内容可读

## 后端测试设计

### 1. 版本对比

#### 用例 1：允许选择两个版本返回字段级差异
- 前置：
  - 已存在同一测试用例的多个版本快照
- 操作：
  - 提交 `testcase_id + base_revision_id + target_revision_id`
- 预期：
  - 返回 `field_diffs`
  - 至少包含：
    - `module`
    - `title`
    - `preconditions`
    - `steps`
    - `expected_results`
    - `priority`
    - `target_type`
    - `automatable`

#### 用例 2：字段未变化时仍返回可识别状态
- 操作：
  - 对比两个只改了单个字段的版本
- 预期：
  - 变化字段 `changed = true`
  - 未变化字段 `changed = false`
  - 前端不需要自行猜测差异

#### 用例 3：版本编号非法时明确报错
- 操作：
  - 传不存在的 `base_revision_id` 或 `target_revision_id`
- 预期：
  - 明确返回错误
  - 不允许 silent fallback 到最近版本

### 2. 结构化审核原因

#### 用例 4：审核通过支持记录原因类型
- 操作：
  - 更新审核状态到 `approved`
  - 传 `review_reason_type`
  - 可选 `review_comment`
- 预期：
  - `Testcase.latest_review_reason_type` 更新成功
  - `Testcase_review_log.review_reason_type` 落库

#### 用例 5：驳回必须同时满足原因类型和驳回说明规则
- 操作：
  - 更新审核状态到 `rejected`
  - 缺少 `review_comment` 或缺少 `review_reason_type`
- 预期：
  - 返回明确错误
  - 不允许只写状态不写原因

#### 用例 6：列表结果包含最新审核原因类型
- 操作：
  - 查询测试用例分页列表
- 预期：
  - 每条记录包含：
    - `latest_review_reason_type`
    - `latest_review_reason_type_label`

### 3. 标签体系

#### 用例 7：单条测试用例可新增标签
- 操作：
  - 编辑某条测试用例标签
  - 提交多个标签名
- 预期：
  - 标签可被创建并与测试用例关联
  - 重复标签不会重复建关联

#### 用例 8：单条测试用例可移除标签
- 操作：
  - 提交新的标签集合，不包含旧标签
- 预期：
  - 多余标签关联被移除
  - 不影响其他测试用例的同名标签

#### 用例 9：列表支持按标签筛选
- 前置：
  - 已存在多个标签和多条测试用例
- 操作：
  - 按单个标签筛选
- 预期：
  - 只返回命中标签的测试用例

### 4. 批量增强

#### 用例 10：支持批量打标
- 操作：
  - 传多条 `testcase_ids`
  - `action_type = add_tags`
- 预期：
  - 所有成功记录都增加目标标签
  - 返回成功数、失败数、失败项

#### 用例 11：支持批量移除标签
- 操作：
  - `action_type = remove_tags`
- 预期：
  - 对应标签关联被移除
  - 未命中的标签不应报致命错误

#### 用例 12：支持批量审核增强
- 操作：
  - `action_type = update_review_status`
  - 同时传 `review_reason_type` 和 `review_comment`
- 预期：
  - 审核状态、审核意见、原因类型同时生效
  - 审核记录完整写入

#### 用例 13：批量操作部分失败时必须返回失败项
- 前置：
  - `testcase_ids` 中包含不存在或无权限记录
- 预期：
  - 返回：
    - `success_count`
    - `failed_count`
    - `failed_items`
  - 不允许静默吞失败

## 前端测试设计

### 1. 标签体验

#### 用例 14：列表页可展示标签摘要
- 预期：
  - 标签显示在次级信息区或摘要区
  - 不挤占主表格 7 列

#### 用例 15：列表页支持按标签筛选
- 操作：
  - 选择一个标签进行筛选
- 预期：
  - 列表只显示命中的测试用例

#### 用例 16：编辑抽屉支持标签编辑
- 操作：
  - 在编辑抽屉新增、删除标签
- 预期：
  - 保存后列表同步更新

### 2. 历史与版本对比

#### 用例 17：历史抽屉仍保留版本列表与审核记录
- 预期：
  - 第一刀已有的历史信息仍可查看
  - 不因加入对比能力而丢失

#### 用例 18：历史抽屉支持两版对比
- 操作：
  - 选择两个版本
- 预期：
  - 页面展示字段级 `before / after`
  - 差异字段有明显标识

#### 用例 19：版本不足两条时对比入口不可误用
- 前置：
  - 只有一条或没有版本快照
- 预期：
  - 页面明确提示不可对比
  - 不允许点了没反应

### 3. 审核增强

#### 用例 20：审核弹窗支持原因类型选择
- 操作：
  - 审核通过或驳回
- 预期：
  - 可选原因类型
  - 驳回时如果规则要求必填，页面有前置校验

#### 用例 21：列表页显示最新审核原因类型中文文案
- 预期：
  - 页面不显示英文编码
  - 中文文案放在次级信息区，不挤占主表格核心列

### 4. 批量增强

#### 用例 22：批量打标交互完整
- 操作：
  - 勾选多条记录
  - 打开批量操作弹窗
  - 选择打标
- 预期：
  - 成功后标签摘要更新

#### 用例 23：批量移除标签交互完整
- 操作：
  - 勾选多条记录
  - 选择移除标签
- 预期：
  - 成功后标签摘要更新

#### 用例 24：批量审核增强交互完整
- 操作：
  - 勾选多条记录
  - 选择批量审核
  - 填写原因类型和审核意见
- 预期：
  - 成功后审核状态和原因同步更新

## 回归测试设计

### 回归 1：正式编辑不回归
- 继续验证单条编辑、版本递增、风险提示。

### 回归 2：Excel 导出不回归
- 继续验证导出文件存在
- 列顺序仍为：
  - `序号`
  - `所属模块`
  - `用例标题`
  - `前置条件`
  - `测试步骤`
  - `预期结果`
  - `优先级`

### 回归 3：自动化草稿入口不回归
- 列表页和历史抽屉仍能进入自动化草稿页
- 目标资产跳转仍然可用

### 回归 4：页面滚动与可读性不回归
- 主内容区可纵向滚动
- 长文本按内容换行
- 主表格不再回到固定高度截断模式

## 失败场景设计

### 失败场景 1：标签重复创建导致脏数据
- 预期：
  - 后端去重
  - 前端不重复显示

### 失败场景 2：批量移除标签把不相关测试用例标签误删
- 预期：
  - 只移除当前选中测试用例与目标标签的关联

### 失败场景 3：审核原因类型丢失但状态已更新
- 预期：
  - 这种情况不能算通过
  - 必须整体失败或事务回滚

### 失败场景 4：两版对比返回不稳定字段
- 预期：
  - 只返回约定字段
  - 不把无关时间戳或内部字段混进差异结果

### 失败场景 5：第二刀把第一刀主表格重新挤乱
- 预期：
  - 主表格仍只服务核心设计阅读
  - 标签、审核原因、自动化摘要只能放次级区域

## 最小验证矩阵
- 后端 pytest：
  - 版本对比
  - 结构化审核
  - 标签新增/删除/筛选
  - 批量打标/移除标签/批量审核
  - 第一刀回归
- 前端：
  - `npm.cmd run type:check`
- 实现完成后人工核对：
  - 标签筛选
  - 版本对比
  - 审核原因类型
  - 批量增强
  - 页面滚动与可读性
