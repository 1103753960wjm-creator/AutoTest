## 背景
- `M1` 已经完成 AI Gateway、三模式切换和测试用例草稿生成。
- `M2` 已经完成需求资产、测试用例资产、审核入库和基础管理闭环。

- 一旦 AI 进入正式生产链路，就不能再停留在“能生成就行”的阶段，必须补上：
  - prompt 可管理

  - `l-tester-master/views/ai/ai_service.py` 中 prompt 仍是硬编码拼接
  - 当前没有统一的 AI 调用日志模型
  - 当前没有 AI 调用失败审计页
  - 当前没有最小统计口径，无法回答“最近谁在用、哪种模式成功率更高、哪里失败最多”

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`，本文件只做需求对齐，不作为直接开工信号。
- AI 能力仍然只能通过统一 `AI Gateway` 接入，禁止业务模块自行记录或自行拼 prompt。
- 所有新增文档、说明、代码注释统一使用中文。
- 新增配置统一进入配置层，不得在业务模块散写地址、密钥、模板内容。
- 本阶段优先补治理闭环，不回头改动 `M1 ~ M3` 已稳定的主功能结构。

## 目标
- 让 AI prompt 从硬编码实现，升级成“可配置、可版本化、可回退”的模板资产。
- 让每一次 AI 调用都能记录请求类型、模式、provider、结果状态、耗时和关联业务对象。
- 让系统管理下可以查看 AI 调用日志、失败明细和基础统计。
- 为后续质量评分、失败分析、成本统计、prompt 迭代提供稳定底座。

## 非目标
- 本次不做复杂 prompt 工作流编排。
- 本次不做向量库、RAG、知识库召回治理。
- 本次不做精确 token 成本结算；如果 provider 暂时不给 usage，只记录空值或估算占位。
- 本次不做自动 prompt A/B 实验。
- 本次不做调用队列、异步重试中心。
- 本次不做完整质量评分系统，只做基础统计口径和失败暴露。

## 当前缺口

### 缺口 1：prompt 硬编码
- 当前测试用例生成和自动化草稿生成的 prompt 都写在 `ai_service.py` 里。
- 这会导致：
  - prompt 无法单独调整
  - prompt 版本不可追溯
  - 修改 prompt 必须发版
  - 无法区分“代码改动”还是“模板改动”

### 缺口 2：调用无日志
- 当前调用结果只在业务资产里留下间接痕迹，没有统一 AI 调用日志。
- 这会导致：
  - 不能按时间回看最近调用
  - 不能按模式或 provider 排查问题
  - 不能统计成功率和失败率

### 缺口 3：失败不可审计
- 当前如果 `local_llm / remote_llm` 返回异常，用户只能在当前页面看到失败提示。
- 失败原因、原始错误、关联对象、调用上下文没有统一沉淀。

### 缺口 4：没有最小治理视图
- 当前系统管理只有 `AI配置`。
- 没有：
  - prompt 模板页
  - 调用日志页
  - 基础统计卡片

## 核心流程

### 流程 A：通过模板生成
1. 用户在系统管理中维护 prompt 模板。
2. AI Gateway 根据请求类型读取“当前启用版本”的模板。
3. 平台把业务变量填进模板，生成最终 prompt。
4. 发起模型调用，并落日志。

### 流程 B：查看调用日志
1. 用户进入系统管理下的 AI 调用日志页。
2. 页面支持按：
  - 请求类型
  - 模式
  - provider
  - 状态
  - 时间范围
  查询调用记录。
3. 用户可展开查看：
  - 关联对象
  - prompt 模板版本
  - 错误摘要
  - 结果摘要

### 流程 C：排查失败
1. 某次测试用例生成或自动化草稿生成失败。
2. 日志中记录失败状态、错误摘要、关联对象和时间。
3. 用户在日志页筛出失败记录。
4. 用户查看失败详情，判断是：
  - 配置问题
  - provider 不可达
  - prompt 输出不合法
  - 业务输入不完整

## 关键约束
- prompt 模板必须通过统一入口读取，业务代码不得继续直接内嵌主模板正文。
- 调用日志记录必须在 AI Gateway 内完成，不能要求业务调用方手工补记。
- 日志详情允许脱敏摘要，不允许把敏感密钥写入日志。
- 第一刀优先记录“最小必要信息”，不追求一次把所有 usage 字段做齐。
- 日志页是治理页，不承担重新执行 AI 调用的职责。

## 风险点
- 风险 1：如果 prompt 模板设计过细，第一刀会被模板系统本身拖慢。
- 风险 2：如果调用日志记录过多原始正文，数据库很快变重。
- 风险 3：如果日志记录不在 Gateway 内统一完成，后续很容易漏记。
- 风险 4：如果把统计做成复杂 BI，大概率会偏离当前阶段目标。

## 验收标准
- [ ] 测试用例生成和自动化草稿生成都不再直接依赖硬编码主 prompt
- [ ] 系统中存在可管理的 prompt 模板资产
- [ ] 每次 AI 调用都会落统一日志
- [ ] 日志至少包含请求类型、模式、provider、状态、耗时、模板版本、关联对象
- [ ] 系统管理下可查看 AI 调用日志列表和失败详情
- [ ] 系统管理下可查看最小统计卡片
- [ ] 至少有后端 pytest 和一个前端类型校验覆盖本阶段改动

## 失败场景
- [ ] 修改模板后，AI Gateway 仍然读取旧硬编码 prompt
- [ ] 调用失败但日志未记录
- [ ] 日志里丢失 `requested_mode / effective_mode`
- [ ] 日志无法反查关联需求或草稿
- [ ] 页面能看到日志列表，但无法区分成功和失败
- [ ] provider 返回异常时，错误详情被完全吞掉
