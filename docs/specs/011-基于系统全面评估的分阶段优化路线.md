- “页面长内容无法滚动、表格固定高度看不清”已经过时；当前主内容区滚动和测试用例页可读性已修复。
- “`.env.example` 缺失”已经过时；当前后端已补示例配置。
- “自动化草稿无法精确定位到目标资产”已经过时；当前 API / Web / App 已支持精确定位。

## 总目标
- 让平台从“功能集合”继续升级成“低学习成本、高连贯性”的测试工作流平台。
- 在不大爆炸的前提下，优先解决底座风险、入口混乱和巨石页面问题。
- 先把已经形成的主链路做稳，再逐步统一首页、导航、工作台和治理能力。

## 非目标
- 本轮不做全仓重写。
- 本轮不同时重构 API / Web / App / 云真机所有工作台。
- 本轮不重新设计整套后端架构。
- 本轮不在没有阶段 spec 的情况下直接推进后续阶段实现。

## 分阶段路线

### 阶段 0：底座止血
- 目标：
  - 先修明显错误和高风险底层问题，不动大 UI 结构。
- 当前阶段重点：
  - `storage.ts` 修正存储 API 错误调用
  - `auth.ts` 修正 getter、最小权限链路和静态菜单依赖问题
  - `axios.ts` 收敛 token/user_id 传递方式
  - 清理明显无效 `console.log`
  - 复核路由、配置和最小启动链路
- 当前阶段不做：
  - 不重构主控台
  - 不重做系统管理
  - 不拆 API / Web / App 巨石页面

### 阶段 1：导航、主控台、统一页面框架
- 目标：
  - 让平台先“顺手”。
- 重点：
  - 重做主控台为测试工作台
  - 统一列表页、抽屉、详情、工作区基本布局
  - 清理模板残留入口
  - 收敛顶部标签为“最近访问 / 收藏 / 常用”

### 阶段 2：测试设计中心收口
- 目标：
  - 不重复造测试设计中心，而是在现有基础上补闭环。
- 重点：
  - 需求资产版本与去重策略
  - 需求 -> 用例 -> 自动化 -> 任务 -> 结果 的追踪补链
  - 需求覆盖率与自动化进度视图
  - 执行结果回链到测试用例和需求

### 阶段 3：自动化三大工作台重构
- 目标：
  - 把巨石页逐步拆成真正工作台。
- 顺序：
  1. 接口自动化
  2. Web 管理
  3. App 管理
- 原则：
  - 一次只动一个模块
  - 每个模块按“导航区 / 编辑区 / 运行结果区 / 详情区”拆分

### 阶段 4：云真机、任务管理、结果中心联动
- 目标：
  - 让执行资源和执行结果真正闭环。
- 重点：
  - 设备会话化
  - 任务向导化
  - 统一结果中心
  - 任务 / 设备 / 结果互相可跳

### 阶段 5：系统管理与 AI 治理化
- 目标：
  - 让平台长期可维护。
- 重点：
  - 权限菜单真源统一
  - 系统管理只保留一套逻辑
  - AI 配置页升级为 AI 治理中心
  - Prompt 模板、调用日志、失败审计、基础统计

## 阶段优先级调整
- 基于当前代码状态，推荐的真实执行顺序是：
  1. 阶段 0
  2. 阶段 1
  3. 阶段 2
  4. 阶段 3
  5. 阶段 4
  6. 阶段 5
- 原因：
  - 当前最大阻碍不是功能缺失，而是底座问题和首页/导航的工作流断裂。
  - 测试设计中心已经不是最薄弱环节，因此不再放在第一优先级。
  - 巨石页面重构必须后置到主链稳定之后，否则会边拆边回归。

## 第一阶段建议
- 当前最适合立刻开工的是：`阶段 0：底座止血`
- 原因：
  - 风险真实存在且已被代码核实
  - 改动范围相对可控
  - 对后续阶段都有直接收益
  - 不会和现有测试设计中心主链冲突

## 第一阶段验收基线
- 登录与最小权限链路不回归。
- 菜单、路由、主内容区不回归。
- 用例生成 / 草稿保存 / 资产跳转不回归。
- 测试用例页当前能力不回归。
- 前端 `typecheck` 可通过。
- 后端相关 pytest 可通过。

## 回退方式
- 阶段化推进，每一阶段单独建 spec、单独验收。
- 不允许跨阶段混改。
- 若某阶段实施后不稳定，可仅回退该阶段变更，不影响前一阶段已验收部分。
