# 测试设计

## 背景
- 本文对应 [004-m2-需求与用例中心-第二刀.md](/e:/AutoTest/docs/specs/004-m2-需求与用例中心-第二刀.md)。
- 当前阶段只进入 `TDD`，不进入实现。
- 本文的目标不是描述最终代码，而是提前锁定：
  - 需要证明哪些行为成立
  - 失败时应该如何暴露
  - 最小回归面覆盖到哪里

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本轮只做 `M2` 第二刀，不进入 `M3` 自动化草稿生成。
- 所有文档、说明、代码注释统一使用中文。
- AI 能力仍然只能通过统一 AI Gateway 进入，本轮资产管理页不得直连模型。

## 本轮测试目标
- 证明需求资产和测试用例资产可以独立管理，而不是只能在“用例生成”页临时查看。
- 证明 `module` 可以作为新增业务字段稳定落库、查询、展示。
- 证明 `category` 继续作为“用例类型”保留，不与 `module` 混用。
- 证明 `preconditions / steps / expected_results` 在前端以多行文本编辑、在后端以数组存储时不会错位。
- 证明测试用例编辑场景切到 `vxe-table` 后，键盘导航和 `Ctrl + S` 行为成立。
- 证明现有“生成草稿 -> 人工审核 -> 保存资产”链路不回归。

## 非目标
- 不验证自动化草稿生成。
- 不验证文件上传、文档解析、RAG、知识库。
- 不验证复杂审批流、多级审核流。
- 不要求本轮补齐全量 E2E 自动化。

## 测试分层

### 一、后端 pytest
用于验证字段模型、查询逻辑、状态更新和接口边界。

优先新增或扩展的测试文件：
- `l-tester-master/tests/test_requirement_common.py`
- `l-tester-master/tests/test_testcase_common.py`
- `l-tester-master/tests/test_requirement_view.py`
- `l-tester-master/tests/test_testcase_view.py`

### 二、前端类型与构建校验
用于保证路由、页面、API 封装和表格接入没有静态错误。

本轮最低要求：
- `npm.cmd run type:check`

如依赖和时间允许，再补：
- `npm.cmd run build`

### 三、前端交互核对清单
当前前端仓库没有现成的组件测试基线，也未接入专门的前端单测框架。
因此本轮前端关键交互先采用“实现后手工核对清单”，但清单要在实现前先定死。

### 四、前后端联动检查
用于验证：
- 请求参数结构一致
- 返回字段与页面消费一致
- `module / category` 职责不串位
- 多行文本与数组的映射稳定

## 后端测试设计

### 1. 数据模型与持久化

#### 用例 1：`module` 可正常落库
- 前置：
  - 已初始化测试数据库
- 输入：
  - 新建一条测试用例，带 `module = "登录模块"`
- 预期：
  - 数据库中存在 `module`
  - 查询详情时可以返回 `module`
  - `category` 仍独立存在

#### 用例 2：`category` 不因 `module` 引入而丢失
- 输入：
  - 创建同时带 `module` 和 `category` 的测试用例
- 预期：
  - 两个字段均可保存
  - 返回详情时两个字段值不互相覆盖

#### 用例 3：多行文本按换行拆分为数组
- 输入：
  - `preconditions = "账号已存在\\n账号状态正常"`
  - `steps = "打开登录页\\n输入账号密码\\n点击登录"`
  - `expected_results = "登录成功\\n跳转首页"`
- 预期：
  - 入库前被规范化成数组
  - 空行被过滤
  - 再次查询时返回数组结构

#### 用例 4：空白行和首尾空格被清洗
- 输入：
  - 带空白行、缩进、尾部空格的多行文本
- 预期：
  - 存储结果不包含空字符串
  - 每行文本被 `trim`

### 2. 需求资产分页列表

#### 用例 5：分页参数缺失时使用默认值
- 输入：
  - 不传 `page_num`、`page_size`
- 预期：
  - 接口正常返回默认分页结果

#### 用例 6：分页参数非法时安全降级
- 输入：
  - `page_num = 0`
  - `page_size = -1`
- 预期：
  - 接口不抛未处理异常
  - 使用安全默认值或返回明确错误

#### 用例 7：支持关键词与来源类型筛选
- 输入：
  - `keyword`
  - `source_type`
- 预期：
  - 只返回符合条件的 requirement 资产

### 3. 测试用例分页列表

#### 用例 8：支持按需求、审核状态、目标类型筛选
- 输入：
  - `requirement_id`
  - `review_status`
  - `target_type`
- 预期：
  - 只返回符合条件的 testcase

#### 用例 9：返回字段包含主编辑字段和次级信息
- 预期：
  - 返回字段至少包含：
    - `module`
    - `title`
    - `preconditions`
    - `steps`
    - `expected_results`
    - `priority`
    - `category`
    - `review_status`

### 4. 测试用例审核状态更新

#### 用例 10：允许更新到合法状态
- 输入：
  - `review_status = approved`
  - `review_status = draft`
  - `review_status = rejected`
- 预期：
  - 状态更新成功

#### 用例 11：拒绝非法状态
- 输入：
  - `review_status = done`
- 预期：
  - 后端拒绝保存
  - 返回明确错误信息

#### 用例 12：跨用户或不存在的 testcase 不可更新
- 输入：
  - 不存在的 `testcase_id`
  - 不属于当前用户的 `testcase_id`
- 预期：
  - 返回明确失败
  - 不更新任何数据

## 前端测试设计

### 1. 路由与页面

#### 用例 13：一级菜单可进入“需求资产”
- 预期：
  - 菜单显示正常
  - 页面路由可访问

#### 用例 14：一级菜单可进入“测试用例”
- 预期：
  - 菜单显示正常
  - 页面路由可访问

### 2. 网格组件接入

#### 用例 15：测试用例主编辑区使用 `vxe-table`
- 预期：
  - 相关页面不再用基础 `el-table` 承担主编辑职责
  - `vxe-table` 依赖正确安装并可编译通过

#### 用例 16：主编辑网格可见列严格为 6 列
- 预期：
  - 只显示：
    - `module`
    - `title`
    - `preconditions`
    - `steps`
    - `expected_results`
    - `priority`
  - `category` 不进入主网格可见列

### 3. 键盘导航

#### 用例 17：方向键切换单元格焦点
- 操作：
  - 使用上、下、左、右键
- 预期：
  - 焦点在网格单元格间移动

#### 用例 18：`Enter` / `F2` 进入编辑
- 预期：
  - 当前单元格进入编辑状态

#### 用例 19：`Esc` 退出编辑
- 预期：
  - 当前单元格退出编辑状态
  - 未保存内容行为符合组件预期

### 4. 快捷入库

#### 用例 20：右下角常驻 `一键入库`
- 预期：
  - 按钮在页面右下角持续可见
  - 按钮文案带 `Ctrl + S`

#### 用例 21：`Ctrl + S` 触发入库
- 操作：
  - 页面聚焦在网格内，按 `Ctrl + S`
- 预期：
  - 触发当前页面入库逻辑
  - 浏览器默认保存行为被拦截

#### 用例 22：未通过前置校验时不可入库
- 场景：
  - 必填字段为空
  - `priority` 非法
- 预期：
  - 页面给出明确提示
  - 不发起后端保存请求

### 5. 多行文本与数组映射

#### 用例 23：页面加载数组后显示为多行文本
- 输入：
  - 后端返回数组字段
- 预期：
  - 页面展示成按换行拼接的文本

#### 用例 24：页面编辑多行文本后提交为数组
- 操作：
  - 输入多行文本并保存
- 预期：
  - 请求体中对应字段为数组

## 联动回归设计

### 用例 25：生成页保存资产链路不回归
- 操作：
  - 从“用例生成”页生成草稿
  - 标记审核
  - 保存资产
- 预期：
  - 原有保存逻辑继续可用
  - 新增 `module` 字段不会导致旧草稿保存报错

### 用例 26：资产页修改审核状态后，生成页详情展示一致
- 预期：
  - 同一条 testcase 在不同页面看到的审核状态一致

### 用例 27：需求资产页查看详情时，`module` 与 `category` 语义不混乱
- 预期：
  - 页面可同时区分“所属模块”和“用例类型”

## 失败场景清单

### 主流程失败
- [ ] `vxe-table` 依赖未安装或版本不兼容，页面无法编译
- [ ] 新页面路由接入后，现有菜单或动态路由异常
- [ ] `Ctrl + S` 未被拦截，触发浏览器默认保存

### 数据失败
- [ ] `module` 为空但页面未拦截
- [ ] `priority` 非法但仍被保存
- [ ] 多行文本拆分数组后出现空字符串
- [ ] `module` 与 `category` 保存后互相覆盖

### 协议失败
- [ ] 前端发送 `module`，后端未返回 `module`
- [ ] 前端发送多行文本，后端错误地按字符串原样入库
- [ ] 前端详情页仍按旧字段消费，导致数据显示错位

### 回归失败
- [ ] 现有“用例生成 -> 保存资产”链路报错
- [ ] 现有 requirement/testcase 详情抽屉显示异常
- [ ] 已有 pytest 因字段变更整体失效
