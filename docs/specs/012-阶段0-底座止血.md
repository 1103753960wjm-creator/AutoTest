## 核心问题

### 问题 1：存储工具实现本身不正确
- 当前 `putJSON/getJSON` 调用了浏览器原生不存在的方法。
- 风险：
  - 某些链路一旦使用这些方法会直接运行时报错
  - 后续 AI 或开发继续复用时会持续扩散

### 问题 2：权限 getter 与 state 字段不一致
- `auth.ts` 中 state 是 `buttons`，getter 却读取 `buttonList`。
- 风险：
  - 按钮权限链路不可信
  - 后续接真权限时更容易出现“有数据但拿不到”的假故障

### 问题 3：请求鉴权参数注入方式不稳
- 当前统一拦截器把 `token/user_id` 写入 `config.data`。
- 风险：
  - GET 请求和部分空 body 请求链路不稳
  - 导出、上传、第三方 axios 分支会出现不一致
  - 后续真权限和网关化改造难度上升

### 问题 4：静态菜单真源问题仍未收口
- 当前菜单仍主要从本地 JSON 来。
- 本阶段不要求一步做成完整后端权限系统，但至少要先把“本地假动态菜单”和真实用户链路的边界说明白，并减少明显错误。

## 方案摘要

### 一、修正存储工具
- 统一 `SessionStorage` 和 `LocalStorage` 的 `putJSON/setJSON/getJSON` 实现。
- 补最小容错，避免脏 JSON 直接炸页面。

### 二、修正权限 store
- 修正 `getButtonList` getter。
- 清理或收敛明显错误字段访问。
- 不在本阶段直接重做整套权限系统。

### 三、收敛 token/user_id 注入
- 请求头优先携带 token。
- `user_id` 改为更稳定的统一注入策略，避免散写到 body。
- 导出、上传等特殊分支同步收敛。
- 若现有后端短期仍依赖 body，允许做兼容过渡，但前端侧不再继续扩散“body 注入”。

### 四、清理最小调试残留
- 清理动态路由和基础链路上的明显 `console.log`。
- 不把这一步扩展成全仓日志清洗。

## 风险点
- 登录、动态路由、菜单渲染属于高风险链路，必须保守修改。
- 若 token 传递方式调整不当，可能导致现有接口整体失效。
- 若静态菜单处理过度，可能影响当前演示和开发环境的菜单显示。

## 验收标准
- [ ] 存储工具 JSON 读写链路不再依赖不存在的浏览器方法
- [ ] `auth.ts` getter 与 state 字段一致
- [ ] 前端请求链路不再默认把 `token/user_id` 粗暴塞进 body
- [ ] 动态路由最小链路不回归
- [ ] 用例生成 / 草稿保存 / 资产跳转不回归
- [ ] 测试用例页当前能力不回归
- [ ] 前端 `npm.cmd run type:check` 通过
- [ ] 受影响后端 pytest 通过或明确说明未跑原因

## 回退方式
- 若 token/header 收敛后出现联调问题，允许在后端鉴权层增加兼容读取，不回退整个阶段。
- 若权限链路变更影响菜单显示，优先回退到“修 getter + 保持现状菜单来源”的最小保守版本。
