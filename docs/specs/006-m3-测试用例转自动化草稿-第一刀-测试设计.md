## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本轮测试设计只覆盖 `M3 第一刀`，不提前扩展到批量转换、自动执行、反向同步。
- 所有新增文档统一使用中文。
- 测试优先验证“现有资产体系可接住新能力”，而不是只验证生成接口能返回一段 JSON。

## 本轮测试目标
- 证明只有 `approved` 的测试用例才允许生成自动化草稿。
- 证明平台可以对单条测试用例生成 `api / web / app` 三种结构化草稿。
- 证明自动化草稿会先入独立草稿记录，而不是直接丢进页面临时状态。
- 证明草稿保存后能写入现有目标资产体系：
  - API -> `Api_script`
  - Web -> `Web_menu + Web_script`
  - App -> `App_menu + App_script`
- 证明保存成功后能返回目标资产主键、入口和追溯关系。
- 证明现有测试用例资产页、API / Web / App 现有编辑页不会因为新增能力回归。

## 测试范围

### 后端
- `automation_draft` 模型和公共逻辑
- 自动化草稿生成接口
- 自动化草稿保存接口
- 测试用例到目标资产的追溯查询接口
- 目标资产适配器：
  - API 适配器
  - Web 适配器
  - App 适配器

### 前端
- 测试用例资产页的生成入口
- 自动化草稿页

## 非目标
- 不做浏览器端完整端到端烟测扩展。
- 不做现有 API / Web / App 页面全部交互回归。
- 不做多模型、多环境矩阵。
- 不做视觉层细节验收。

## 用例设计

### 用例 1：未审核测试用例禁止生成草稿
- 前置：
  - 准备一条 `review_status = draft` 或 `rejected` 的测试用例
- 操作：
  - 调用自动化草稿生成接口
- 预期：
  - 后端拒绝请求
  - 返回明确错误信息
  - 不创建任何草稿记录

### 用例 2：API 草稿生成成功
- 前置：
  - 准备一条 `approved` 测试用例
- 操作：
  - 以 `target_type = api` 发起生成
- 预期：
  - 返回结构化 `draft_payload`
  - 返回 `draft_id`
  - `draft_payload` 满足 `Api_script` 基本骨架：
    - `name`
    - `description`
    - `type`
    - `config`
    - `script`
  - 草稿记录写入 `Automation_draft`

### 用例 3：Web 草稿生成成功
- 操作：
  - 以 `target_type = web` 发起生成
- 预期：
  - 返回结构化草稿
  - 步骤结构符合现有 `Web_script.script` 的可消费形态
  - 允许包含定位器占位或建议值

### 用例 4：App 草稿生成成功
- 操作：
  - 以 `target_type = app` 发起生成
- 预期：
  - 返回结构化草稿
  - 步骤结构符合现有 `App_script.script` 的可消费形态
  - 允许元素图片、坐标等为空位

### 用例 5：API 草稿保存成功
- 前置：
  - 已生成一条 API 草稿
- 操作：
  - 提交编辑后的草稿 payload
- 预期：
  - 创建 `Api_script`
  - 草稿记录更新为已保存
  - 返回 `target_asset_id`
  - 返回 API 场景编辑页入口信息

### 用例 6：Web 草稿保存成功
- 前置：
  - 已生成一条 Web 草稿
- 操作：
  - 提交编辑后的草稿 payload
- 预期：
  - 自动确保 `AI生成草稿` 目录存在
  - 创建 `Web_menu`
  - 创建 `Web_script`
  - 返回 `target_menu_id`
  - 返回 Web 编辑页入口信息

### 用例 7：App 草稿保存成功
- 前置：
  - 已生成一条 App 草稿
- 操作：
  - 提交编辑后的草稿 payload
- 预期：
  - 自动确保 `AI生成草稿` 目录存在
  - 创建 `App_menu`
  - 创建 `App_script`
  - 返回 App 编辑页入口信息

### 用例 8：草稿保存后追溯关系正确
- 操作：
  - 查询某条测试用例的自动化草稿记录
- 预期：
  - 可看到目标类型、保存状态、目标资产编号
  - 目标资产可反查到来源测试用例

### 用例 9：前端入口显隐正确
- 前置：
  - 页面列表同时包含 `approved` 和非 `approved` 测试用例
- 预期：
  - `approved` 行显示或启用生成入口
  - 非 `approved` 行禁用生成入口或提示不可操作

### 用例 10：前端保存成功后跳转信息正确
- 操作：
  - 在自动化草稿页保存目标资产
- 预期：
  - 页面收到 `target_route`
  - 可跳转到对应 API / Web / App 编辑页

## 失败场景设计

### 失败场景 1：目标类型非法
- 输入：
  - `target_type = xxx`
- 预期：
  - 后端拒绝
  - 不创建草稿

### 失败场景 2：测试用例不存在
- 输入：
  - 不存在的 `testcase_id`
- 预期：
  - 返回明确错误信息
  - 不创建草稿

### 失败场景 3：草稿 payload 结构不符合目标资产格式
- 输入：
  - 缺失必填字段的编辑 payload
- 预期：
  - 后端拒绝保存
  - 不写入目标资产
  - 草稿保留为未保存状态

### 失败场景 4：Web / App 草稿保存时菜单创建失败
- 预期：
  - 不创建半残的脚本记录
  - 返回明确错误
  - 事务整体回滚

### 失败场景 5：保存成功但未记录追溯关系
- 预期：
  - 测试直接失败
  - 不接受“资产写入成功但追溯缺失”的结果

### 失败场景 6：前端页面刷新后草稿完全丢失
- 第一刀原则：
  - 如未保存，则允许不持久化编辑态
  - 但必须能通过 `draft_id` 重新拉取后端草稿
- 预期：
  - 页面可重新加载草稿记录

## 最小验证矩阵

### 后端 pytest
- 草稿生成服务测试
- 草稿保存服务测试
- API / Web / App 三类适配器测试
- 追溯查询测试
- 接口层测试

### 前端验证
- `vue-tsc`
- 至少一条与生成入口和保存跳转相关的页面级验证

## 通过标准
- 后端 pytest 覆盖三种目标类型的生成与保存主链
- 前端类型校验通过
- 关键失败场景至少覆盖：
  - 非 approved 禁止生成
  - 非法目标类型拒绝
  - 保存失败不产生脏资产
  - 追溯关系必须存在
