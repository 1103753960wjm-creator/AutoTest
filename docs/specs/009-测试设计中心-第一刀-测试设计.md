# 测试设计

## 背景
- 本文对应 [009-测试设计中心-第一刀.md](/e:/AutoTest/docs/specs/009-测试设计中心-第一刀.md)。
- 当前阶段只进入 `TDD`，不进入实现。
- 本文目标是提前锁定：
  - 需要证明哪些设计行为成立
  - 失败时应该如何暴露
  - 最小回归面应该覆盖到哪里

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`。
- 本轮只覆盖“测试设计中心第一刀”，不进入 `M4` 治理实现。
- 所有新增文档、说明、代码注释统一使用中文。
- 本轮必须保护现有 `M3` 自动化草稿入口和追溯摘要，不允许因为测试设计改造而回归。
- 英文编码可以继续存在于内部存储，但主界面和导出结果必须以中文业务文案为准。

## 本轮测试目标
- 证明正式测试用例支持单条编辑，而不是只能改审核状态。
- 证明正式测试用例编辑后会生成版本快照，并且 `version` 递增。
- 证明审核动作会留下审核意见、驳回原因和审核记录。
- 证明测试用例资产页支持批量基础修改。
- 证明生成后的正式测试用例页主表格按固定顺序展示核心设计字段。
- 证明主界面不再直接显示 `happy_path`、`negative_path`、`general` 这类英文编码。
- 证明正式测试用例页支持导出 Excel，且导出列顺序与页面主表格一致。
- 证明已有关联自动化草稿的测试用例在内容变更后会出现明确风险提示。

- 不验证版本 diff 和版本回滚。
- 不验证场景链路、前后置依赖。

## 测试分层

### 一、后端 pytest
用于验证：
- 正式测试用例编辑
- 审核记录落库
- 版本快照
- 批量操作
- 列表结果增强字段

优先新增或扩展的测试文件：
- `l-tester-master/tests/test_testcase_common.py`
- `l-tester-master/tests/test_testcase_view.py`

### 二、前端类型校验
用于保证：
- 页面重构后类型不破
- API 封装与页面消费字段一致
- 表格导出和中文映射逻辑没有静态错误

本轮最低要求：
- `npm.cmd run type:check`

### 三、前端交互核对清单
由于当前前端没有成熟组件单测基线，本轮关键交互先采用实现后核对清单：
- 主表格列顺序
- 中文映射显示
- 编辑抽屉行为
- 批量操作行为
- 导出 Excel 行为

### 四、浏览器烟测候选
如果本轮实现后需要固化高风险链路，可追加一条浏览器烟测覆盖：
- 查询正式测试用例
- 打开编辑
- 保存
- 查看审核记录
- 导出 Excel

## 后端测试设计

### 1. 正式测试用例编辑

#### 用例 1：允许编辑正式测试用例核心字段
- 前置：
  - 已存在一条正式测试用例
- 操作：
  - 提交编辑请求，修改：
    - `module`
    - `title`
    - `preconditions`
    - `steps`
    - `expected_results`
    - `priority`
    - `target_type`
    - `automatable`
- 预期：
  - 数据更新成功
  - 返回更新后的正式测试用例
  - requirement 关联不丢失

#### 用例 2：编辑后版本号递增
- 前置：
  - 已存在 `version = 1` 的正式测试用例
- 操作：
  - 提交一次有效编辑
- 预期：
  - 保存后 `version = 2`

#### 用例 3：编辑前先写入版本快照
- 前置：
  - 已存在一条正式测试用例
- 操作：
  - 提交编辑请求
- 预期：
  - 生成一条 `Testcase_revision`
  - `snapshot` 保存编辑前内容
  - `version` 与快照版本对应

#### 用例 4：编辑原因为空时的处理符合约定
- 操作：
  - 不传 `edit_reason` 或传空字符串
- 预期：
  - 如果第一刀要求非必填，则允许保存并写默认摘要
  - 如果第一刀要求必填，则明确报错
- 说明：
  - 实现前必须把这一条约定写死，不能模糊处理

### 2. 审核动作与审核记录

#### 用例 5：审核通过会写入审核记录
- 操作：
  - 把测试用例状态从 `draft` 改到 `approved`
  - 传 `review_comment`
- 预期：
  - `Testcase.review_status` 更新成功
  - `latest_review_comment` 更新
  - 新增一条 `Testcase_review_log`

#### 用例 6：驳回必须填写驳回原因
- 操作：
  - 把测试用例状态改为 `rejected`
  - 不传 `review_comment`
- 预期：
  - 后端拒绝保存
  - 返回明确错误信息

#### 用例 7：审核记录能按时间返回
- 操作：
  - 连续进行多次审核状态变更
- 预期：
  - 记录按时间倒序或约定顺序返回
  - 能看到 `from_status`、`to_status`、`review_comment`

### 3. 批量基础修改

#### 用例 8：支持批量修改优先级
- 操作：
  - 传多条 `testcase_ids`
  - `action_type = update_priority`
- 预期：
  - 成功记录优先级被统一修改
  - 返回 `success_count`

#### 用例 9：支持批量修改目标类型
- 操作：
  - `action_type = update_target_type`
- 预期：
  - 成功记录目标类型被统一修改

#### 用例 10：支持批量修改所属模块
- 操作：
  - `action_type = update_module`
- 预期：
  - 成功记录所属模块被统一修改

#### 用例 11：支持批量修改审核状态
- 操作：
  - `action_type = update_review_status`
  - 合法状态值
- 预期：
  - 成功记录审核状态被统一修改
  - 必要时写入审核记录

#### 用例 12：部分失败时返回失败项
- 前置：
  - `testcase_ids` 中包含不存在或无权限记录
- 预期：
  - 返回：
    - `success_count`
    - `failed_count`
    - `failed_items`
  - 不允许静默吞失败

### 4. 列表结果增强

#### 用例 13：列表结果包含最新审核意见摘要
- 操作：
  - 查询测试用例分页列表
- 预期：
  - 每条记录包含：
    - `latest_review_comment`
    - `latest_review_time`
    - `revision_count`
    - `history_available`

#### 用例 14：已有自动化草稿的测试用例返回同步风险提示字段
- 前置：
  - 测试用例已存在自动化草稿
  - 再次编辑正式测试用例
- 预期：
  - 列表结果或详情结果中存在风险提示字段
  - 前端可据此展示“自动化草稿可能过期”

## 前端测试设计

### 1. 主表格重排

#### 用例 15：正式测试用例页主表格顺序固定
- 预期：
  - 主表格按以下顺序展示：
    - `序号`
    - `所属模块`
    - `用例标题`
    - `前置条件`
    - `测试步骤`
    - `预期结果`
    - `优先级`

#### 用例 16：治理字段不再占主表格核心阅读区
- 预期：
  - `用例类型`
  - `目标类型`
  - `所属需求`
  - `自动化进度`
  - `审核状态`
  不再占用主表格核心设计列区

### 2. 中文映射

#### 用例 17：页面不直接显示英文编码
- 前置：
  - 后端仍返回：
    - `happy_path`
    - `negative_path`
    - `required_field_validation`
    - `permission_or_visibility`
    - `general`
- 预期：
  - 页面显示中文业务文案
  - 不直接显示原始英文编码

#### 用例 18：类型和目标类型映射正确
- 预期：
  - `general -> 通用`
  - `api -> 接口`
  - `web -> Web`
  - `app -> App`

### 3. 编辑交互

#### 用例 19：可以打开单条编辑面板
- 操作：
  - 点击“编辑”
- 预期：
  - 打开编辑抽屉或侧边面板
  - 表单回填当前正式测试用例内容

#### 用例 20：保存后列表即时刷新
- 操作：
  - 编辑后保存
- 预期：
  - 主表格内容更新
  - 版本号相关摘要可刷新

#### 用例 21：已有关联自动化草稿时显示风险提示
- 前置：
  - 当前用例已有自动化草稿或目标资产
- 预期：
  - 编辑页或列表页有明确提示
  - 用户不会误以为自动化资产自动同步

### 4. 批量操作

#### 用例 22：支持多选并触发批量操作栏
- 操作：
  - 多选若干行
- 预期：
  - 页面出现批量操作入口

#### 用例 23：批量修改成功后反馈明确
- 预期：
  - 显示成功数
  - 如有失败，显示失败数与失败摘要

### 5. Excel 导出

#### 用例 24：支持导出 Excel
- 操作：
  - 点击“导出 Excel”
- 预期：
  - 浏览器开始下载文件
  - 文件格式为 Excel

#### 用例 25：导出表头顺序与页面一致
- 预期：
  - Excel 表头顺序与主表格一致：
    - `序号`
    - `所属模块`
    - `用例标题`
    - `前置条件`
    - `测试步骤`
    - `预期结果`
    - `优先级`

#### 用例 26：导出内容使用中文业务文案
- 前置：
  - 数据中存在英文编码
- 预期：
  - 导出文件展示中文文案
  - 不直接导出 `happy_path`、`general` 这类原始编码到用户可见列

## 失败场景设计

### 失败场景 1：正式测试用例编辑成功，但版本号没变
- 预期：
  - 测试直接失败

### 失败场景 2：内容变更了，但没有版本快照
- 预期：
  - 测试直接失败

### 失败场景 3：驳回成功，但没有驳回原因
- 预期：
  - 测试直接失败

### 失败场景 4：批量修改部分失败，但接口返回全成功
- 预期：
  - 测试直接失败

### 失败场景 5：主界面仍然直接显示英文编码
- 预期：
  - 页面级验证失败

### 失败场景 6：导出 Excel 表头顺序与页面不一致
- 预期：
  - 导出验证失败

### 失败场景 7：已有关联自动化草稿的测试用例被修改，但没有同步风险提示
- 预期：
  - 页面级验证失败
