# 规格说明

## 背景
- `M1` 已经完成了平台级 AI 配置和测试用例草稿生成。
- `M2` 已经完成了需求资产、测试用例资产、审核入库和资产管理闭环。
- 当前最大的缺口不再是“能不能生成和保存测试用例”，而是“审核通过后的测试用例如何进入现有 API / Web / App 自动化体系”。
- 如果这一段不打通，平台仍然只是“会生成测试用例的资产平台”，还不是“测试生产平台”。
- 当前仓库里三类执行资产都已经存在成熟模型和页面：
  - API：`Api_script`
  - Web：`Web_menu + Web_script`
  - App：`App_menu + App_script`
- 所以 `M3` 的正确方向不是平行再造一套执行体系，而是把“测试用例 -> 自动化草稿”接到现有资产体系里。

## 当前生效规则
- 继续遵守 `Spec -> TDD -> Execution`，本文件只做需求对齐，不作为直接开工信号。
- 所有 AI 能力仍然必须走统一 `AI Gateway`，不得在 API / Web / App 模块中散接模型调用。
- 所有新增文档、设计说明、代码注释统一使用中文。
- 新增配置统一进入配置层，不得在业务模块硬编码地址、密钥和路径。
- 优先复用现有目录、接口封装、返回结构和脚本保存方式，不平行复制一套新后台。

## 目标
- 让“已审核通过”的测试用例可以生成 API / Web / App 自动化草稿。
- 草稿必须先可查看、可编辑、可确认，再进入目标资产体系。
- 保存后的自动化草稿必须能落到现有资产模型中，而不是停留在临时页面状态。
- 平台必须保留测试用例与目标自动化资产之间的追溯关系。
- 为后续“执行前人工修订”和“自动化资产治理”提供稳定入口。

## 非目标
- 本次不做“一键生成后直接执行”。
- 本次不做批量多用例同时转换。
- 本次不做自动生成后立即挂入定时任务。
- 本次不做复杂审批流、版本对比、冲突合并。
- 本次不要求 API 草稿自动精准绑定真实接口定义。
- 本次不要求 Web / App 草稿自动生成完全可运行的稳定定位器。
- 本次不做反向同步，即不要求目标脚本编辑后自动回写测试用例。

## 核心流程

### 流程 A：从测试用例生成自动化草稿
1. 用户进入“测试用例资产”页面。
2. 用户选择一条 `review_status = approved` 的测试用例。
3. 用户选择目标类型：`api / web / app`。
4. 平台通过统一入口生成自动化草稿。
5. 页面展示结构化草稿，允许人工继续修改。

### 流程 B：确认并保存到目标资产体系
1. 用户在草稿页确认内容。
2. 用户点击“保存到目标资产”。
3. 后端把草稿转换成现有目标资产模型需要的结构。
4. 平台写入目标资产表，并返回目标资产 `id`、名称和入口信息。
5. 前端提示保存成功，并支持跳转到对应目标页面继续细化。

### 流程 C：追溯
1. 用户从测试用例详情可以查看已生成的自动化草稿记录。
2. 用户可以看到：
  - 目标类型
  - 生成模式
  - 保存状态
  - 对应目标资产编号
3. 后续报告、运行、维护时，能从目标资产反查到来源测试用例。

### 追溯约束
- 每条自动化草稿都必须能反查来源测试用例。
- 每条已保存的自动化草稿都必须记录目标资产主键。
- 同一条测试用例可以对应多条自动化草稿，但目标类型不同或版本不同要分记录保存。

## 目标资产适配策略

### API 草稿
- 目标模型：
  - `Api_script`
- 适配结果结构：
  - `name`
  - `description`
  - `type`
  - `config`
  - `script`
- 第一刀约束：
  - 先生成“测试场景骨架”，允许步骤里出现“待绑定接口”占位。
  - 不要求本阶段自动完成真实接口节点绑定。
  - 用户后续进入现有 API 场景页继续补齐接口选择、环境和参数。

### Web 草稿
- 目标模型：
  - `Web_menu`
  - `Web_script`
- 适配结果结构：
  - 使用现有 `Web_script.script` 的 JSON 结构
  - 步骤类型先收敛为：打开、点击、输入、等待、断言
- 第一刀约束：
  - 定位器允许先生成占位值或建议值。
  - 必须让用户在保存前可见并可改。

### App 草稿
- 目标模型：
  - `App_menu`
  - `App_script`
- 适配结果结构：
  - 使用现有 `App_script.script` 的 JSON 结构
  - 步骤类型先收敛为：启动、点击、输入、等待、断言、截图
- 第一刀约束：
  - 元素图片、坐标、设备特定信息允许保留空位。
  - 不要求保存后立即可在所有设备上直接执行。

## 保存策略

### API
- 直接创建一条 `Api_script` 记录。
- 脚本名称默认使用：
  - `[AI草稿][API]{测试用例标题}`
- 保存成功后返回现有 API 场景页入口信息。

### Web
- 自动确保存在统一草稿目录：
  - `AI生成草稿`
- 在目录下创建对应菜单节点和 `Web_script` 记录。
- 默认名称：
  - `[AI草稿][WEB]{测试用例标题}`

### App
- 自动确保存在统一草稿目录：
  - `AI生成草稿`
- 在目录下创建对应菜单节点和 `App_script` 记录。
- 默认名称：
  - `[AI草稿][APP]{测试用例标题}`

## 页面落点

### 测试用例资产页
- 每条已审核测试用例增加：
  - `生成 API 草稿`
  - `生成 Web 草稿`
  - `生成 App 草稿`
- 对 `review_status != approved` 的测试用例禁用生成入口。

### 自动化草稿页
- 页面负责：
  - 展示来源测试用例
  - 展示生成告警
  - 展示结构化草稿
  - 允许保存到目标资产
  - 保存成功后跳转目标编辑页
- 第一刀不单独做复杂列表页，以“当前生成、即时确认”为主。

## 关键约束
- 只允许从 `approved` 测试用例生成自动化草稿。
- 第一刀一次只处理一条测试用例、一个目标类型。
- API / Web / App 模块不得直接自己调用模型，只能消费统一生成结果。
- 保存到目标资产前必须经过人工确认。
- 第一刀允许生成“可编辑但未完全可执行”的草稿，不强行追求一次生成即稳定运行。
- 草稿页必须明确展示“AI 生成仅为初稿，仍需人工校对”。

## 风险点
- 风险 1：如果 API 草稿试图直接绑定真实接口，范围会迅速失控。
- 风险 2：如果 Web / App 强求第一刀就生成稳定定位器，成功率会很差，且难以解释失败原因。
- 风险 3：如果不建立独立草稿记录，保存前修改内容无法追溯。
- 风险 4：如果直接写入目标资产却不记录来源，后续治理和回溯会断链。
- 风险 5：如果同时做批量转换、多版本对比、反向同步，会拖慢核心闭环。

## 验收标准
- [ ] 用户可以从测试用例资产页对单条 `approved` 测试用例发起自动化草稿生成
- [ ] 用户可以选择目标类型：`api / web / app`
- [ ] 平台可以返回结构化草稿，而不是纯文本描述
- [ ] 用户可以在保存前查看和编辑草稿
- [ ] API 草稿可保存为 `Api_script`
- [ ] Web 草稿可保存为 `Web_menu + Web_script`
- [ ] App 草稿可保存为 `App_menu + App_script`
- [ ] 保存成功后，前端能拿到目标资产编号和入口信息
- [ ] 平台能保留测试用例与目标资产之间的追溯关系
- [ ] 至少有后端 pytest、前端类型校验覆盖本阶段首刀改动

## 失败场景
- [ ] 测试用例未审核通过时仍能发起生成
- [ ] 目标类型非法时仍被后端接受
- [ ] 草稿生成成功但未保存时，页面刷新后完全丢失且无任何提示
- [ ] 保存到目标资产时，目标模型写入结构不符合现有页面消费格式
- [ ] Web / App 草稿保存时未能创建对应菜单节点
- [ ] 保存成功后，返回值缺少目标资产入口信息
- [ ] 追溯记录缺失，导致无法从测试用例查看已生成资产
- [ ] 现有 API / Web / App 编辑页无法打开新生成的草稿资产

## 下一步实现边界
- 下一步实现只做 `M3 第一刀`：
  - 单条测试用例生成单目标类型草稿
  - 自动化草稿记录入库
  - 手工确认后保存到现有目标资产
  - 返回目标资产入口
- 暂不做：
  - 批量转换
  - 自动执行
  - 反向同步
  - 复杂资产治理页
