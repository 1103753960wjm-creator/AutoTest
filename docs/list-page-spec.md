# 阶段 1 - 列表页统一规范

## 1. 目标

本轮只统一列表页的结构规则，不统一所有表格字段和业务逻辑。

第一轮优先统一：

- 页面头部接入方式
- 筛选区布局
- 状态态接入方式
- 表格区与分页区的结构位置
- 主操作区与批量动作区的位置表达

## 2. 结构规则

### 2.1 页面头部

- 页面头部始终由 Layout 挂载的 `PlatformPageHeader` 承接
- 页面通过 `route meta + usePlatformPageHeader` 注入标题、副描述、主操作和辅助信息
- 页面内部不得再保留旧 `.page-header`

### 2.2 页面主体

推荐骨架：

1. 筛选区
2. 列表区或状态态
3. 分页区
4. 对话框区

其中：

- 筛选区优先复用 `FilterBar`
- 表格区、分页区和弹窗区在结构上分层
- 对话框区可保留原业务实现，不在本轮重写

### 2.3 筛选区

- 优先承接关键词搜索、常用筛选项和重置按钮
- 更复杂的高级筛选可暂缓，只统一基础布局
- 搜索与筛选动作尽量落在 `FilterBar` 主区和 actions 区

### 2.4 状态态

- 优先复用现有 `ui-states`
- 必须区分：
  - `loading`
  - `request-error`
  - `forbidden`
  - `empty`
  - `search-empty`

说明：

- `empty` 表示系统本身没有数据
- `search-empty` 表示筛选后无结果

### 2.5 表格区

- 保留原业务字段、列定义和接口逻辑
- 不为统一而重写表格逻辑
- 若页面有批量动作，先统一位置，不统一能力

### 2.6 分页区

- 分页放在表格区之后的固定位置
- 位置和对齐方式尽量一致
- 原分页参数和请求逻辑保留

## 3. 第一批试点页

本轮试点仅选 3 个项目列表页：

1. 测试设计项目列表
2. Web 自动化项目列表
3. App 自动化项目列表

理由：

- 都是高频入口页
- 都是典型列表页
- 数据层风险较低
- 能覆盖多个模块

## 4. 批量动作边界

- 批量动作区本轮只统一位置
- 原有批量逻辑继续保留
- 没有批量动作的页面不硬补

## 5. 弹窗边界

- 新建、编辑、详情弹窗保留原实现
- 本轮只做“列表主体骨架”和“弹窗区分层”
- 不重写弹窗体系

## 6. 第一批试点的基准页

测试设计项目列表作为结构模板基准页：

- 学结构，不搬逻辑
- Web / App 项目列表应参考其结构骨架
- 但保留各自原有请求逻辑、字段逻辑和弹窗逻辑

## 7. 暂缓范围

- 复杂工作台页
- 深链接 / 回跳规则
- 复杂批量操作统一
- 全站一次性替换
- 列表页之外的模块内部详情页、编辑页和工作区
