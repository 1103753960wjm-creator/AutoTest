# TestHub 开发流程规则

本文定义当前仓库的默认开发流程。当前项目按 `Spec -> TDD -> Execution` 的顺序执行开发，目标是先对齐需求，再约束实现，再做回归验证。

## 0. Spec 优先原则
- 默认流程不是“先想方案就写代码”，而是先把需求、边界、验收标准对齐清楚。
- 没有 spec 对齐，不进入正式开发。
- spec 不要求一定是长文档，但至少要明确以下内容：
  - 目标：这次改动到底要解决什么问题
  - 边界：改哪里，不改哪里
  - 输入输出：页面行为、接口字段、数据结构、状态变化
  - 验收标准：改完之后如何证明功能成立
  - 风险点：JWT 认证、WebSocket、Celery 任务、执行器、通知、报告、媒体文件、AI 调用是否受影响
- 如果需求不完整，先列“已知事实、待确认项、当前假设”，再继续后续设计。
- 对以下改动，spec 是强制项：
  - 新功能
  - 大改或跨模块改动
  - 接口协议变更
  - 数据库模型或迁移变更
  - AI 能力接入、统一配置调整、执行链路改造
  - JWT / Celery / Channels / Allure / 通知链路改造

## 1. TDD 优先原则
- 在 spec 对齐后，默认先明确行为和失败场景，再写实现，再做回归验证。
- 对新功能、跨模块改动、高风险链路，优先写测试设计或验证清单，再改业务代码。
- TDD 约束的是开发顺序和验收方式，不是让人为了凑测试去做脱离现状的重构。
- 如果现有项目条件不允许完整先写自动化测试，至少先写“验证目标 + 影响面 + 最小回归面”，再实施代码改动。

## 2. 默认工作方式
- 先读代码，再下手，不根据名字臆测实现。
- 优先做最小可落地改动，不顺手重构无关区域。
- 先判断改动落点：只改 Django、只改 Vue，还是前后端联动。
- 高风险链路默认需要更强验证：登录鉴权、AI 模型配置、Browser-use、Selenium/Playwright/Airtest、定时任务、通知告警、执行结果和报告。

## 3. Fast Path
适用于同时满足以下条件的小改动：
- 只影响一个子系统：仅前端或仅后端。
- 只改一个模块，且不改数据库模型、不改接口协议、不改 Celery / Channels / JWT。
- 需求边界清晰，不需要额外 spec 澄清。
- 预计 30 分钟内可完成实现与验证。

执行步骤：
1. 读取相关入口和调用链。
2. 先写最小 spec 摘要，明确“这次改什么、不改什么、怎么验收”。
3. 再写最小验证目标，明确“改完怎么证明是对的”。
4. 再实现最小改动。
5. 运行最贴近改动面的校验。
6. 说明改动点、验证结果、未验证项。

## 4. Standard Path
满足以下任一条件时，必须走标准流程：
- 同时改前端和后端。
- 改数据库模型、迁移、定时任务、执行器、通知、鉴权、WebSocket。
- 改接口字段、分页结构、认证结构或前端消费协议。
- 涉及历史结果、报告、媒体路径、状态流转。
- 改动跨两个以上模块。
- 需求不清晰，需要先收敛范围。

执行步骤：
1. Discovery：明确入口、调用链、数据结构、风险点。
2. Spec Alignment：先写需求摘要、范围边界、验收标准、失败场景、回退方式。
3. Test Design：先写测试设计、验证清单或失败场景列表。
4. Design：决定改动边界，先收敛范围，再写代码。
5. Execution：按模块顺序修改，避免一边写一边扩散。
6. Verification：按影响面做针对性验证。
7. Reporting：给出结果、风险、后续建议。

## 5. 代码变更前必须确认的事
- 当前实现到底以哪个文件为入口。
- 当前页面、接口、任务或配置是否被别处复用。
- 当前字段是否已落库、是否进了前端表格、日志、报告或 WebSocket 推送。
- 当前逻辑是否带有定时任务、通知、文件落盘、副作用。
- 这次需求有没有完成最小 spec 对齐。

## 6. 验证规则
- 后端改动：至少做导入级、`python manage.py check`、调用级或 `pytest` 级验证；能跑受影响用例就跑局部。
- 前端改动：至少做 `npm run build` 或 `npm run lint` 级验证；能做页面相关检查就跑受影响范围。
- 前后端联动：至少检查请求参数、响应结构、页面展示三者一致。
- 涉及 Celery / Channels / 执行器：至少检查状态更新、日志输出、结果落库或推送链路。
- 无法验证时必须明确说明原因，不能假装已经验证。

## 7. TDD 最低门槛
- 新增后端业务接口：优先补 `pytest`、`manage.py test` 或最小调用级验证。
- 新增前端关键交互：优先补构建校验，条件允许时补组件/页面级验证。
- 改登录、权限、AI、调度、执行器、通知、报告：必须先列失败场景，再改实现。
- 没有自动化测试条件，不等于可以跳过测试设计。

## 8. 推荐验证命令
- 后端优先：`python manage.py check`、`pytest`、受影响模块导入检查、必要时 `python manage.py test <app>`。
- 前端优先：`npm run build`、`npm run lint`、必要时受影响页面手工核对。
- 不要求每次都跑全量，但必须说明为什么只跑局部。

## 9. 数据与协议变更规则
- 改 Django `models.py` 时，同时检查 serializer、viewset、urls、迁移说明和前端消费逻辑。
- 改接口返回结构时，同时检查 `frontend/src/api/*`、`frontend/src/utils/api.js` 和页面调用方。
- 改 token、用户信息、权限字段时，同时检查 `frontend/src/stores/user.js`、`frontend/src/router/index.js` 和后端 SimpleJWT 链路。
- 改执行状态、报告路径、截图视频等字段时，同时检查数据库、媒体目录、前端展示和通知内容。

## 10. 文档更新规则
- 小改动不强制新建文档。
- 当改动涉及架构边界、目录职责、全局约束时，应同步更新 `.antigravity` 对应规则文件。
- 当改动涉及需求边界、产品行为、验收标准时，应同步更新本次任务的 spec 或需求摘要。
- 所有新增或修改的 spec、设计文档、验收清单、任务摘要、交付说明统一使用中文。
- 所有新增或修改的代码注释统一使用中文；字段名、协议关键字、库名等技术标识可保留英文原文。
- 不为了流程补无价值文档。

## 11. Spec 模板
用于新功能、大改、跨模块改动、接口协议变更、数据库变更、AI 能力接入。

```md
# Spec

## 背景
- 当前问题：
- 触发原因：

## 目标
- 本次要解决：

## 非目标
- 本次明确不做：

## 范围
- 受影响模块：
- 受影响页面：
- 受影响接口/模型：

## 输入输出
- 输入：
- 输出：
- 状态变化：

## 方案摘要
- 关键改动：
- 关键约束：

## 风险点
- 风险 1：
- 风险 2：

## 验收标准
- [ ] 行为 1 成立
- [ ] 行为 2 成立
- [ ] 不引入回归

## 回退方式
- 如何关闭/回退：
```

## 12. 失败场景模板
用于 TDD 设计、风险梳理和高风险链路改造前的失败清单。

```md
# Failure Scenarios

## 主流程失败
- [ ] 输入缺失时如何表现
- [ ] 依赖服务不可用时如何表现
- [ ] 权限不足时如何表现

## 数据失败
- [ ] 字段为空
- [ ] 字段类型不匹配
- [ ] 历史数据兼容失败

## 链路失败
- [ ] 前后端字段不一致
- [ ] 状态未同步
- [ ] 日志/结果未落库

## 回归失败
- [ ] 旧功能 A 是否受影响
- [ ] 旧功能 B 是否受影响
```

## 13. 验收清单模板
用于实现完成后的交付核对。

```md
# Acceptance Checklist

## Spec
- [ ] 需求边界已对齐
- [ ] 非目标已明确

## Code
- [ ] 改动只落在预期模块
- [ ] 未引入无关重构
- [ ] 配置未散落硬编码

## Verification
- [ ] 后端验证已执行
- [ ] 前端验证已执行
- [ ] 联动验证已执行
- [ ] 未验证项已说明

## Delivery
- [ ] 改动点已说明
- [ ] 风险已说明
- [ ] 后续建议已说明
```

## 14. 禁止事项
- 不做 spec 对齐就直接实现。
- 不读调用链就直接重写。
- 为了“看起来更规范”大规模搬目录。
- 发现旧代码风格不统一时顺手全仓替换。
- 用新抽象覆盖旧问题，但不补验证。
