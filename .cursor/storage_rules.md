# TestHub 存储与访问规则

## 1. 文件职责

本文件属于 B 层“项目规则”，用于约束当前仓库中的数据访问、配置访问、文件落盘和客户端存储方式。

## 2. 前端访问链路

统一链路：

`View -> src/api/* -> src/utils/api.js -> Backend /api/*`

必须遵守：

- 页面和组件不得直接发裸请求。
- 鉴权头、token 刷新、401 处理统一沿用 `frontend/src/utils/api.js`。
- 仅将前端运行时必需信息保存在本地存储，如 token、用户基础信息、界面状态；真实业务数据不得长期依赖浏览器缓存代替后端真源。

## 3. 后端访问链路

统一链路：

`backend/urls.py -> apps/<module>/urls.py -> views -> serializers/services/models`

必须遵守：

- 接口入口只做收参、调度、权限控制和响应组织。
- 数据库存取优先通过 Django ORM。
- 多步骤流程、第三方依赖调用、执行器链路和跨对象状态更新应进入服务层、执行器或工具层。

## 4. 数据库存储规则

- 新增或修改模型时，优先修改对应 `models.py`，并同步检查 `serializers.py`、`views.py`、关联查询和前端消费端。
- 不强推全仓统一为 `code/message/data`；新增或改动接口应保持所在模块既有响应风格和兼容性。
- 若必须使用原生 SQL，应有明确性能或兼容性理由，并限制在局部封装中。

## 5. 文件与媒体落盘规则

- 上传物、截图、录像、报告、Allure 结果等运行产物统一进入 `media` 路径体系。
- 日志产物统一进入 `logs`。
- Allure 工具、插件和配置统一留在 `allure`。
- 新增运行产物时，应保证目录结构与业务结果可追溯，不新增散落临时目录。

## 6. 配置访问规则

- 后端环境配置统一优先通过 `backend/settings.py` 和 `decouple.config` 收敛。
- 新增后端配置时，不在业务代码中继续扩散 `os.getenv`、地址常量和硬编码密钥；如遇历史散点，新增改动应优先向集中配置收敛。
- 前端环境配置统一通过 Vite 环境变量和现有配置入口管理，不在页面内散写环境判断。
- AI 模型配置、提示词配置、行为配置优先走现有数据库配置或统一配置入口，不新增一套平行配置通路。

## 7. 客户端状态与认证规则

- 用户 token、刷新 token 和过期时间统一沿用 `frontend/src/stores/user.js` 与 `frontend/src/utils/api.js` 管理。
- 登录、刷新、退出相关变更时，必须同步检查本地存储、Pinia 状态、请求拦截器和路由跳转。

## 8. 外部服务访问规则

- Webhook、邮件、AI 服务、SSE / WebSocket、执行器依赖等外部能力应通过现有模块和配置入口接入。
- 敏感配置不得写死到新增代码、规则文档或示例中。

## 9. 明确禁止

- Vue 页面绕过 `src/api/*` 直接访问接口。
- 在接口入口层散写文件路径拼接和运行产物目录逻辑。
- 同一业务数据既依赖数据库又依赖浏览器缓存，并靠猜测决定谁是真源。
- 为了单次开发方便继续扩散硬编码地址、密钥和环境变量读取。
