# TestHub 技术栈与代码风格规则

## 1. 文件职责

本文件属于 B 层“项目规则”，用于定义当前仓库长期有效的代码风格、命名习惯、接口兼容策略和交付底线。

## 2. 后端代码风格

- Python 文件名、模块名、函数名、变量名优先使用 `snake_case`。
- Django / DRF 入口优先保持短小清晰，复杂逻辑下沉。
- 新增日志优先进入现有日志体系，不到处散写 `print`。
- 新增字段、模型和序列化逻辑时，优先复用现有 app 的组织方式。

## 3. 前端代码风格

- Vue 页面和组件优先沿用当前单文件组件写法。
- 页面级逻辑留在页面内，共享逻辑再考虑提取到 `stores`、`components` 或 `utils`。
- 与后端交互统一通过 `frontend/src/api/*`，不要在组件里直接请求。
- 长列表页、长表单页和结果页必须显式检查滚动容器、内容截断和主内容区展示，不依赖浏览器默认行为碰运气。

## 4. 命名与结构

- Django app 保持按业务域拆分，如 `projects`、`testcases`、`requirement_analysis`、`ui_automation`、`app_automation`。
- 前端页面目录与业务域尽量对应，便于从页面反查接口和后端模块。
- 新增 API 封装函数名应尽量与后端语义一致，避免过度抽象命名。

## 5. 接口与字段兼容规则

- 新增或修改接口时，优先保持所在模块既有返回结构和错误风格，不强推全仓统一包装。
- 页面字段名优先与后端语义一致；如需适配，优先在前端 API 层或页面适配层处理。
- 若修改公共字段、路由 meta、认证字段、来源标签、处理状态等共享语义，必须同步检查消费端和相关文档。

## 6. 编码与中文规则

- 所有规则文件、文档文件和源码文件统一使用 UTF-8。
- 当前仓库存在历史中文内容；遇到乱码时先判断是文件编码问题还是终端输出问题，再决定是否修复源码。
- 与用户沟通、规范文档、设计文档、交付说明统一使用中文。
- 所有新增或修改的代码注释统一使用中文；字段名、协议关键字、库名、框架名和第三方服务名可保留英文原文。

## 6.1 注释策略（不强制数量，强调必要性与质量）

- 不强制“写代码必须写注释”；禁止为了凑注释而写显而易见的描述性注释（例如“获取列表”“设置变量”“返回结果”）。
- 当出现以下任一场景时，必须补充**中文注释**，说明“为什么/约束/取舍/不能改什么”：
  - 非直观业务规则或边界条件（默认值、特殊校验、异常分支为何如此处理）
  - 兼容性与历史包袱（旧接口/旧数据/旧行为的兼容原因与风险）
  - 性能/安全/一致性取舍（为何不用更直观写法、为何要缓存/限制/降级）
  - 明显副作用或高风险点（会触发异步任务/通知/报告/状态流转/缓存刷新等）
- 注释语言要求沿用 6 节：注释用中文，关键字/库名/第三方服务名可保留英文原文。

## 7. 验证与交付底线

- Python 改动至少做导入级、编译级、调用级或测试级验证。
- 前端改动至少做构建级、类型级或页面级验证。
- 高风险改动必须说明改了什么、为什么安全、验证了什么、还剩什么未验证项。
- 每次完成显著功能、规则或文档变更后，应同步更新 `更新日志.md`。

## 8. 明确禁止

- 为了统一风格把老代码一次性重构成完全不同的体系。
- 在前端复制后端业务规则，在后端复制前端展示规则。
- 未经验证就调整登录、token、动态路由、执行器、异步任务、报告或 AI 调用链。
