# TestHub 开发流程规则

## 1. 文件职责

本文件属于 B 层“项目规则”，用于定义当前仓库的默认开发流程、验证要求和文档同步要求。

本文件不负责定义目录架构、存储路径和长期技术约束；这些分别由 `architecture.md`、`storage_rules.md`、`project_rules.md` 负责。

## 2. 默认交付流程

当前仓库默认按以下顺序执行：

1. `Spec`
2. `TDD`
3. `Execution`

开始编码前，至少要明确：

- 当前生效规则
- 范围边界
- 验收标准
- 失败场景或验证目标

## 3. Spec 对齐要求

未完成 spec 对齐前，不进入正式实现。

### 3.0 不确定性处理（Spec 阶段禁止擅自决定）

在 Spec 阶段，凡是存在不确定/歧义/多种可选方案且会影响实现或验收的内容，**必须先询问用户并等待答复**，禁止 AI 自己决定后继续推进（包括但不限于进入 TDD 或 Execution）。

典型必须提问的“不确定项”示例：

- 需求语义不清：要“优化/修复/兼容/重构”到底指什么、优先级如何
- 范围边界不清：改哪里/不改哪里、是否涉及前后端联动、是否允许改接口/模型
- 行为细节不清：边界值处理、默认值、排序/分页规则、空态/错误态 UI 表现
- 协议与字段不清：请求字段、返回结构、错误码/错误信息、字段命名与兼容策略
- 验收标准不清：如何判定完成、需要哪些验证/截图/日志/回归点
- 风险取舍不清：安全与便捷、性能与一致性、兼容旧数据与引入新字段之间的取舍

允许 AI 自主推进的前提：

- 不确定项仅影响“文案措辞/排版微调”等非关键行为，且不改变既有语义；否则一律先问再动。

### 3.1 Spec 确认闸门（默认强制，非“小修小改”不得跳过）

当任务满足“必须做明确 spec”的任一条件（见 3 节列表），或不满足 8 节“快速路径”条件时，必须按以下顺序推进：

1. AI 输出一份可审核的 Spec（可用 9 节模板，允许是精简版）。
2. **暂停**，等待用户明确回复 **“Spec 已确认 / 可以进入 TDD”**（或等价表述）。
3. 收到确认后，才允许进入 4 节 TDD（写验证目标、失败场景、必要测试设计）。
4. 完成 TDD 后，**再次暂停**，等待用户明确回复 **“可以进入 Execution / 开始实现”**（或等价表述），才允许进入正式编码实现。

说明：
- 若用户仅补充了 spec 的部分信息但未明确确认，则视为仍处于 Spec 阶段，应继续补齐并等待确认。
- 用户可以要求“快速路径”，但前提仍是满足 8 节条件；否则不得用“快”绕过确认闸门。

spec 不要求一定是长文档，但至少要明确：

- 目标：这次改动要解决什么问题
- 范围：改哪里，不改哪里
- 输入输出：页面行为、接口字段、数据结构、状态变化
- 验收：改完如何证明成立
- 风险：登录、权限、执行器、异步任务、通知、报告、WebSocket 等是否受影响

以下场景必须做明确 spec：

- 新功能
- 跨模块改动
- 接口协议变更
- 数据模型变更
- 认证、权限、执行、通知、报告链路改动
- AI 接入方式或配置入口改动

## 4. TDD 最低门槛

TDD 约束的是开发顺序和验证口径，不要求为凑测试而脱离现状重构。

### 4.1 TDD 交付物（必须先写出来）

进入 Execution 前，至少要产出以下内容（写在对话里即可，不强制落盘）：

- 验证目标：这次改动要“证明成立”的 3-7 条断言（可覆盖接口/页面/数据/副作用）
- 失败场景：至少覆盖“权限/状态不允许、依赖失败/超时、参数边界、并发/重复提交”中与本次相关的项
- 验证层级选择：本次用哪一层来验证（见 4.2），以及为什么够用
- 最小验证清单：你将实际执行的检查（命令/接口调用/页面操作），以及通过标准

### 4.2 验证层级选择（优先从高性价比开始）

按优先级从上到下选择，尽量用“更贴近改动面且成本更低”的验证方式：

- 接口级：DRF view/serializer 的请求响应、权限、分页、过滤、错误码与返回结构
- 服务级：services/utils 的业务分支、状态流转、副作用编排
- 序列化/校验级：serializer 字段校验、默认值、跨字段校验、只读/必填约束
- 模型级：模型方法、约束、信号、关系与查询条件（不以“跑得过迁移”为验证）
- 前端页面级：关键交互路径、表单校验、列表刷新、错误提示、加载态与空态
- 构建/静态级：lint、构建、类型/语法检查（只能作为补充，不足以替代关键行为验证）

### 4.3 不同改动类型的最低验证要求（更具体）

- 新增/修改后端接口（含字段、权限、过滤、排序、分页、返回结构）：
  - 必须：接口级验证目标 + 至少 1 条可执行的调用级验证（能写 `pytest` 就写；否则明确用什么方式调用验证）
  - 必须：列出典型错误输入与权限失败的返回表现（状态码/错误信息/前端如何展示）
- 新增/修改前端关键交互（提交、删除、批量、导入导出、状态流转入口、编辑器保存）：
  - 必须：页面级验证目标 + 至少 1 条受影响页面的手工验证步骤（含预期 UI 反馈）
  - 必须：覆盖重复点击/并发提交/网络失败/接口报错下的 UI 表现
- 触达高风险链路（登录/JWT/权限、执行器、异步任务、通知、报告、WebSocket、AI 配置/调用）：
  - 必须：失败场景列表（至少 5 条）+ 回退/止损方式
  - 必须：说明“状态/结果/日志/前端展示”四个面是否受影响，以及各自怎么验证

### 4.4 TDD 模板（建议直接复制填空）

```md
## 验证目标（3-7 条）
- [ ] 目标 1：
- [ ] 目标 2：

## 失败场景（按本次相关项列）
- 权限/状态不允许：
- 参数边界：
- 依赖失败/超时：
- 并发/重复提交：

## 选择的验证层级
- 本次采用：接口级 / 服务级 / 页面级 / …
- 原因：为什么足够覆盖风险

## 最小验证清单（我会实际执行什么）
- 命令/操作 1：
  - 通过标准：
- 命令/操作 2：
  - 通过标准：
```

### 4.5 底线

- 没有自动化测试条件，不等于可以跳过测试设计；至少要把“验证目标 + 失败场景 + 最小验证清单”写清楚。
- 若你要求我进入 Execution，我会先检查 4.1 的交付物是否齐全；不齐全则继续停留在 TDD 阶段补齐。

## 5. Execution 规则

- 先确认真实入口和调用链，再实施改动。
- 优先做最小闭环改动，不顺手重构无关区域。
- 前后端联动问题先判断落点：只改前端、只改后端，还是两边一起改。
- 高风险链路默认需要更强验证和更明确的回退说明。

### 5.1 修 Bug / 小优化的硬性要求（防回归清单）

当任务目标是“修 bug/修兼容/体验小优化”时，在进入 Execution 前必须显式检查并说明（写在对话里即可）：

- 边界值：空值/缺省值/极值/长文本/分页边界/重复提交等，本次哪些相关、如何处理
- 回归面：会影响哪些旧入口/旧接口/旧字段/旧页面/旧数据，至少列出 3-5 个需要关注的点
- 周边模块影响：是否会改变状态流转、权限判断、缓存/刷新时机、异步任务触发、日志/通知/报告等副作用
- 禁止顺手大重构：除非 bug 根因明确要求重构且已在 Spec 里写清范围与回退，否则不得把修复扩展成“整块重写/全链路重构”

以上任一项不清楚，回到 Spec 阶段先询问用户澄清，禁止凭经验直接拍板推进。

## 6. 验证规则

- 后端改动：至少做导入级、编译级、调用级或 `pytest` 级验证。
- 前端改动：至少做构建级、类型级或受影响页面验证。
- 前后端联动：至少核对请求参数、响应结构、页面展示三者一致。
- 无法验证时必须明确说明原因，不能假装已经验证。

## 7. 文档同步规则

以下内容发生变化时，应同步更新对应真源：

- 项目长期规则变化：更新 `.antigravity/*.md`
- 阶段边界、验收口径、下一步主线变化：更新 `docs/project-memory/current_phase.md`
- 新对话启动摘要变化：更新 `docs/project-memory/dialogue_bootstrap.md`
- 显著功能、规则或文档变更：更新 `更新日志.md`

## 8. 快速路径与标准路径

适合快速路径的场景：

- 只影响一个子系统
- 不改接口协议和数据模型
- 边界清晰
- 能在短时间内完成实现和验证

### 8.1 “小修小改”（快速路径）判定口径

同时满足以下条件，可视为“小修小改”，允许在同一回合内完成 Spec + 最小验证目标 + Execution（但仍不得跳过最小验证目标）：

- 改动范围极小：主要改动集中在 **单一文件**（允许顺带改 1 个紧邻的测试/样式/文案文件，但总计不超过 2 个文件），且不涉及跨模块联动
- 不触达高风险链路：不涉及认证/权限/JWT、执行器、异步任务、通知、报告、WebSocket、AI 调用/配置入口；也不改 `frontend/src/utils/api.js`、路由守卫、Pinia 用户/鉴权相关 store
- 不改协议与真源：不改后端接口字段/返回结构、不新增/修改接口、不改数据模型/迁移；前端不改 `frontend/src/api/*` 的接口封装函数签名与返回适配逻辑
- 不引入行为变化：只做明确的 bug 修复/展示微调/文案修正/样式修正；不新增页面流转、交互分支、复杂状态机或新的权限判断
- 可快速验证：能在本地用 1-2 条命令或 1 个页面操作完成验证（否则视为非“小修小改”）

只要以上任一条不满足，则不得走快速路径，必须回到 3.1 的“Spec 确认闸门”流程。

执行顺序：

1. 读取相关入口和调用链
2. 写最小 spec 摘要
3. 写最小验证目标
4. 实施改动
5. 运行贴近改动面的校验
6. 说明改动、验证和未验证项

满足以下任一条件时，应走标准路径：

- 同时改前端和后端
- 改模型、认证、权限、执行器、通知、异步任务、报告
- 改接口字段或返回结构
- 改动跨两个以上模块
- 需求不清晰，需要先收敛范围

## 9. Spec 最小模板

以下模板用于快速对齐需求，不要求每次都写成长文，但结构应尽量完整：

```md
# Spec

## 背景
- 当前问题：
- 触发原因：

## 目标
- 本次要解决：

## 非目标
- 本次明确不做：

## 范围
- 受影响模块：
- 受影响页面 / 接口 / 模型：

## 输入输出
- 输入：
- 输出：
- 状态变化：

## 风险点
- 风险 1：
- 风险 2：

## 验收标准
- [ ] 行为 1 成立
- [ ] 行为 2 成立
- [ ] 不引入明显回归

## 回退方式
- 如何关闭、回退或止损：
```

## 10. 测试设计与测试用例编写方法

测试设计应从 spec 直接展开，先定覆盖面，再决定自动化还是手工验证。

最少要覆盖以下几类测试点：

- 主流程：正常输入下，核心目标能否成立
- 边界条件：空值、极值、缺省值、重复提交、分页边界、长文本等
- 异常流程：依赖失败、接口报错、超时、后端异常、第三方不可用
- 权限与状态：未登录、权限不足、状态不允许、资源不存在
- 状态流转：创建、更新、删除、执行、取消、完成等状态是否闭环
- 回归影响：旧页面、旧接口、旧字段、旧入口是否被破坏

编写测试用例时，建议按以下顺序组织：

1. 先写“验证目标”，明确这条用例想证明什么。
2. 再写“前置条件”，明确环境、登录态、数据依赖和开关条件。
3. 再写“输入或操作步骤”，保持可复现。
4. 最后写“预期结果”，分别覆盖接口结果、数据变化、页面展示和副作用。

按改动类型选择验证方式：

- 后端改动：优先写接口级、调用级、序列化级或模型级验证，条件允许时补 `pytest`。
- 前端改动：优先写页面交互验证、构建验证和关键渲染路径验证。
- 前后端联动：至少同时验证请求参数、响应结构、页面展示和状态同步。

## 11. 验收清单模板

实现完成后，至少按以下清单做一次交付核对：

```md
# Acceptance Checklist

## Spec
- [ ] 目标和非目标已对齐
- [ ] 范围边界已明确
- [ ] 风险点已说明

## Test Design
- [ ] 主流程已覆盖
- [ ] 边界和异常已覆盖
- [ ] 回归影响已检查

## Execution
- [ ] 改动只落在预期模块
- [ ] 未引入无关重构
- [ ] 配置未继续散写

## Verification
- [ ] 已执行后端验证
- [ ] 已执行前端验证
- [ ] 已执行联动验证或说明未执行原因
```
